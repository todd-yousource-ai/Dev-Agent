# Forge

An enterprise security platform that enforces policy-driven access, data handling, and service trust controls across internal systems and integrations.

## What It Does
Forge provides a layered enforcement model for enterprise applications, services, and developer workflows. It protects sensitive operations and data by combining architecture standards, interface contracts, implementation guidance, and testable subsystem boundaries. The platform is intended for teams building and operating security-sensitive services that must conform to documented trust, integration, and enforcement rules.

## Key Subsystems
- Architecture Standards — defines the system-wide trust model, layering rules, and enforcement boundaries.
- Interface Contracts — specifies wire formats, API expectations, and integration constraints between components.
- Decision Records — captures architectural decisions and the rationale behind security-critical tradeoffs.
- Conventions — standardizes implementation patterns so generated and human-written code follow the same rules.
- TRDs — describe subsystem behavior, constraints, and security requirements at the technical design level.
- PRDs — define product and operational requirements that drive platform capabilities.
- Dev Agent Output — generated implementation in `src/` that must conform to the standards and specs.
- Test Suite — validates subsystem behavior and conformance to documented architecture and interfaces.

## Architecture Overview
Forge follows a spec-first enforcement model: requirements are defined in TRDs and PRDs, constrained by architecture standards, then realized in generated implementation under `src/` and verified by tests. The effective trust boundary starts with documented architecture rules in `forge-standards/`, flows through interface and subsystem contracts, and ends in code and CI validation. This repository is organized so reviewers and implementation agents can trace every behavior back to a governing design document.

## Repository Structure
```text
forge-docs/          — source TRDs and PRDs
forge-standards/     — architecture, interfaces, decisions, conventions
CLAUDE.md            — LLM coding instructions (read this first)
src/                 — implementation (generated by Forge Dev Agent)
tests/               — test suite
.github/workflows/   — CI
```

## Getting Started
- Read `CLAUDE.md` first; it defines how contributors and coding agents should work in this repo.
- Review `forge-standards/ARCHITECTURE.md` and `forge-standards/INTERFACES.md` before changing implementation details.
- Use `forge-docs/` as the source of truth for subsystem behavior, requirements, and constraints.
- Keep changes in `src/` aligned with the documented standards and add or update tests in `tests/`.
- Check `.github/workflows/` to understand CI expectations before opening a PR.
