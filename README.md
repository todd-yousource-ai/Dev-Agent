# Forge

A policy-enforcement security platform that applies architecture-level trust controls, interface governance, and subsystem isolation so enterprise systems can ship safely without relying on ad hoc application defenses.

## What It Does
Forge enforces security through standardized architecture rules, explicit interface contracts, and repository-scoped implementation constraints. It protects enterprise services and developer workflows by making trust boundaries, subsystem responsibilities, and protocol behavior first-class parts of the build process. The platform is designed for teams building governed systems where enforcement must be consistent across specs, code, and CI.

## Key Subsystems
- **Architecture Standards** — defines the system trust model, enforcement boundaries, and permitted composition patterns.
- **Interface Standards** — governs wire formats, API contracts, and cross-subsystem protocol expectations.
- **Decision Records** — captures architectural decisions and enforcement rationale so implementations stay aligned.
- **Conventions** — standardizes repository, implementation, and documentation expectations for contributors and generated code.
- **LLM Coding Instructions** — constrains agent behavior so generated changes follow project architecture and repo rules.
- **Implementation Layer (`src/`)** — contains the platform code generated against the documented standards and TRDs.
- **Test Suite (`tests/`)** — verifies behavioral correctness and enforcement compliance.
- **CI Workflows** — automates validation so architectural and interface rules are checked continuously.

## Architecture Overview
Forge uses a documentation-first enforcement model: TRDs define required behavior, `forge-standards/` defines the architectural and interface rules, and implementation in `src/` must conform to both. The trust model is specification-driven and repository-enforced, with coding agents and human contributors constrained by `CLAUDE.md`, then validated through tests and CI. In practice, architecture rules are established first, interfaces are fixed second, and implementation is expected to remain downstream of those decisions.

## Repository Structure
```text
forge-docs/          — source TRDs and PRDs
forge-standards/     — architecture, interfaces, decisions, conventions
CLAUDE.md            — LLM coding instructions (read this first)
src/                 — implementation (generated by Crafted Dev Agent)
tests/               — test suite
.github/workflows/   — CI
```

## Getting Started
- Read `CLAUDE.md` before making changes; it defines how contributors and coding agents are expected to work in this repo.
- Review `forge-standards/ARCHITECTURE.md` and `forge-standards/INTERFACES.md` to understand the trust model, enforcement order, and contract boundaries.
- Use `forge-docs/` as the source of truth for product and technical requirements before changing implementation.
- Keep changes in `src/` aligned with the documented standards and verify behavior in `tests/`.
- Run or inspect `.github/workflows/` when modifying contracts, architecture-sensitive code, or validation logic.
