# Forge

A policy-enforced enterprise security platform that turns architecture standards and technical requirements into governed implementation, interface, and verification workflows.

## What It Does
Forge defines and enforces how security-critical platform components are specified, implemented, and integrated across the repository. It protects architectural integrity by binding subsystem behavior to Technical Requirement Documents (TRDs), interface contracts, and architecture rules stored in `forge-docs/` and `forge-standards/`.  
For developers, reviewers, and automation agents, it provides a single governed path from requirements to code in `src/`, with tests and CI validating conformance.

## Key Subsystems
- **TRD / PRD Source Set** — defines the required platform behavior, capabilities, and security constraints.
- **Architecture Standards** — enforces structural rules, design decisions, and cross-cutting conventions.
- **Interface Contracts** — defines wire formats, API contracts, and subsystem boundaries.
- **LLM Development Guardrails** — constrains code generation and edits through `CLAUDE.md`.
- **Implementation Layer** — realizes the platform in `src/` according to the approved specs.
- **Test Suite** — verifies conformance of implementation against requirements and interfaces.
- **CI Workflows** — automates validation, regression detection, and repository policy checks.

## Architecture Overview
Forge uses a spec-first enforcement model: requirements originate in TRDs/PRDs, architecture constraints are codified in `forge-standards/`, and implementation in `src/` must conform to both. The trust model is document-driven — developers and coding agents are expected to read and follow `CLAUDE.md`, architecture standards, and interface definitions before changing code.  
Enforcement proceeds from requirements and standards to implementation, then through tests and GitHub Actions CI for validation.

## Repository Structure
```text
forge-docs/          — source TRDs and PRDs
forge-standards/     — architecture, interfaces, decisions, conventions
CLAUDE.md            — LLM coding instructions (read this first)
src/                 — implementation (generated by Crafted Dev Agent)
tests/               — test suite
.github/workflows/   — CI
```

## Getting Started
- Read `CLAUDE.md` first; it defines repository-specific coding and agent behavior.
- Review `forge-standards/ARCHITECTURE.md` and `forge-standards/INTERFACES.md` before touching implementation.
- Use `forge-docs/` as the source of truth for subsystem behavior, constraints, and acceptance criteria.
- Make changes in `src/`, then add or update coverage in `tests/`.
- Use CI in `.github/workflows/` as the final conformance check before merge.
