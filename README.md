# Forge Security Platform

A policy-enforced enterprise security platform that applies architectural, interface, and implementation constraints consistently across generated and human-authored code.

## What It Does
Forge defines and enforces a standards-first development model for security-sensitive systems: requirements live in TRDs/PRDs, architecture and interface contracts live in `forge-standards/`, and implementation in `src/` must conform to those decisions. It protects platform integrity by making trust boundaries, wire contracts, and subsystem responsibilities explicit before code is written. The repository is designed for developers, reviewers, and LLM-based coding agents working on enterprise security components under a shared enforcement model.

## Key Subsystems
- Architecture Standards — defines system structure, trust boundaries, and composition rules.
- Interface Contracts — specifies wire formats and API-level behavioral contracts.
- Decision Records — captures architectural choices and enforcement constraints.
- Conventions Layer — standardizes implementation patterns used across generated code.
- Dev Agent Instructions — constrains LLM coding behavior through `CLAUDE.md`.
- Implementation Surface — `src/` contains code generated or authored under Forge standards.
- Test Suite — validates conformance of implementation to architecture and interface rules.
- CI Workflows — enforces automated checks on changes before integration.

## Architecture Overview
The platform follows a specs-first enforcement model: TRDs and PRDs define required behavior, `forge-standards/` translates that into architecture and interface constraints, and only then does implementation in `src/` proceed. Trust is anchored in documented standards rather than ad hoc code paths, so reviews and CI can verify conformance in a predictable order: requirements, architecture, interfaces, implementation, tests. This keeps generated and manual changes aligned under the same security and design rules.

## Repository Structure
```text
forge-docs/          — source TRDs and PRDs
forge-standards/     — architecture, interfaces, decisions, conventions
CLAUDE.md            — LLM coding instructions (read this first)
src/                 — implementation (generated by Forge Dev Agent)
tests/               — test suite
.github/workflows/   — CI
```

## Getting Started
- Read `CLAUDE.md` first; it defines how code should be written and constrained in this repository.
- Review `forge-standards/ARCHITECTURE.md` to understand the system model, trust boundaries, and enforcement rules.
- Check `forge-standards/INTERFACES.md` before changing any APIs, message shapes, or integration behavior.
- Use `forge-docs/` as the source of truth for product and technical requirements in the TRDs/PRDs.
- Keep implementation in `src/` and tests in `tests/` aligned with the documented standards and decisions.
