# Enterprise Security Platform

A policy-driven security enforcement platform that protects enterprise infrastructure through layered subsystem controls, cryptographic trust validation, and continuous compliance monitoring.

## What It Does

This platform enforces security policies across enterprise environments by coordinating specialized subsystems that handle authentication, authorization, secrets management, audit logging, and runtime threat detection. It provides a unified enforcement layer that ensures every request—whether from a user, service, or automated agent—is validated against organizational policy before accessing any protected resource. The architecture is designed for zero-trust environments where every interaction must be explicitly verified and continuously audited.

## Key Subsystems

- **Policy Engine** — evaluates and enforces access control policies across all subsystems using a declarative rule framework
- **Identity & Authentication** — manages credential lifecycle, multi-factor enforcement, and cryptographic identity verification
- **Authorization Gateway** — enforces fine-grained, attribute-based access decisions at the request path level
- **Secrets Vault** — handles generation, rotation, and secure distribution of secrets with automatic expiry enforcement
- **Audit & Compliance Logger** — captures immutable, tamper-evident logs of all enforcement decisions and policy changes
- **Runtime Threat Detection** — monitors live traffic for anomalous patterns and triggers automated containment responses
- **Certificate & Trust Manager** — manages PKI infrastructure, certificate issuance, and chain-of-trust validation
- **Configuration Enforcement** — ensures infrastructure and service configurations conform to declared security baselines

## Architecture Overview

The platform follows a layered enforcement model where every inbound request passes through the Identity, Authorization, and Policy Engine subsystems in strict sequential order before reaching any downstream service. Trust is established through cryptographic verification at each layer boundary—no subsystem implicitly trusts another. The Audit Logger operates as a sidecar to all enforcement points, ensuring that every decision is recorded regardless of outcome, and the Runtime Threat Detection system operates independently to provide defense-in-depth monitoring.

## Getting Started

- **Read `CLAUDE.md` first** — it contains the coding rules, naming conventions, and constraints that every contributor (human or LLM) must follow before writing any code.
- **Review `forge-standards/ARCHITECTURE.md`** to understand the enforcement order, trust boundaries, and how subsystems communicate.
- **Check `forge-standards/INTERFACES.md`** for wire formats, API contracts, and expected request/response schemas before implementing any integration point.
- **Find detailed requirements in `forge-docs/`** — each TRD specifies the exact behavior, edge cases, and acceptance criteria for its subsystem.
- **Run the existing test suite** (`tests/`) before submitting changes to ensure you haven't broken enforcement invariants.
