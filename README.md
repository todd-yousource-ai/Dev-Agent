# Forge Platform

A unified enterprise security enforcement platform that guarantees policy compliance, access control, and audit integrity across all system interactions through a layered trust architecture.

## What It Does

Forge enforces security policies at every layer of the stack — from API gateway ingestion through policy evaluation, session management, and audit logging — ensuring that no request reaches a protected resource without passing through the full enforcement chain. It provides centralized identity resolution, attribute-based access control, cryptographic audit trails, and real-time policy evaluation for enterprise environments where compliance and zero-trust principles are non-negotiable.

## Key Subsystems

- **Gateway Enforcer** — ingress validation, rate limiting, and request normalization at the edge
- **Identity Resolver** — unified identity resolution across federated providers with token introspection
- **Policy Engine** — attribute-based access control (ABAC) evaluation with policy versioning and conflict resolution
- **Session Manager** — stateful session lifecycle management with cryptographic binding and revocation
- **Audit Ledger** — append-only, tamper-evident audit trail with hash-chained event records
- **Secret Vault** — encryption key management, secret rotation, and secure credential distribution
- **Compliance Monitor** — continuous compliance posture assessment against configurable regulatory frameworks
- **Event Bus** — internal pub/sub for cross-subsystem enforcement signals and policy change propagation

## Architecture Overview

All requests flow through a strict enforcement order: Gateway Enforcer → Identity Resolver → Policy Engine → target resource, with the Audit Ledger recording every decision at each stage. The trust model is hierarchical — no downstream subsystem trusts upstream claims without independent verification, and all inter-subsystem communication uses mutually authenticated TLS with short-lived credentials issued by the Secret Vault. The Event Bus enables reactive enforcement: a revoked session or updated policy propagates to all enforcement points within the consistency window.

## Getting Started

- **Read `CLAUDE.md` first** — it contains the coding rules, naming conventions, and enforcement constraints that all code (human or LLM-authored) must follow.
- **Review `forge-standards/ARCHITECTURE.md`** to understand the enforcement order, trust boundaries, and how subsystems compose.
- **Check `forge-standards/INTERFACES.md`** for wire formats, API contracts, and inter-subsystem protocol definitions before writing any integration code.
- **Find the spec for your subsystem** in `forge-docs/` — each TRD defines the exact behavior, error handling, and compliance requirements.
- **Run `make test`** before submitting — CI enforces full test passage and lint compliance on every PR.
