# Forge Security Platform

A policy-driven enterprise security platform that enforces identity, device, network, and data controls in one place to prevent unauthorized access and reduce lateral movement across internal systems.

## What It Does

Forge Security Platform applies consistent enforcement across authentication, authorization, service-to-service communication, and protected resource access. It protects enterprise applications, internal APIs, and operator workflows by combining policy evaluation, trust-boundary enforcement, and standards-based interfaces. The platform is built for engineering, platform, and security teams that need centralized control without embedding custom security logic into every service.

## Key Subsystems

- Identity and Access Control — enforces authentication, session validation, and access policy decisions for users and services.
- Policy Engine — evaluates security rules and determines allow, deny, and conditional enforcement outcomes.
- Service-to-Service Trust — enforces authenticated internal communication and trust establishment between platform components.
- API Enforcement Layer — protects exposed interfaces and applies request-level security controls before business logic executes.
- Audit and Event Logging — records security-relevant decisions, access attempts, and enforcement outcomes for traceability.
- Secrets and Key Management — protects signing keys, credentials, and other sensitive security material used by platform services.
- Administrative Control Plane — governs operator actions, configuration changes, and privileged workflows.
- Standards and Interface Contracts — defines shared protocols, wire formats, and implementation constraints across subsystems.

## Architecture Overview

The platform follows a centralized-policy, distributed-enforcement model: control-plane components define policy and trust rules, while enforcement points apply them at the edge of every protected interaction. Requests are authenticated first, evaluated against policy second, and only then allowed to reach protected services or data paths. Architecture and interface constraints are defined in the repository standards so generated and hand-written components remain consistent.

## Getting Started

- Read `CLAUDE.md` first; it defines repo-specific implementation and contribution rules.
- Review `forge-standards/ARCHITECTURE.md` to understand component boundaries, trust model, and enforcement order.
- Review `forge-standards/INTERFACES.md` before changing APIs, schemas, or message formats.
- Use `forge-docs/` as the source of truth for product requirements, technical requirements, and subsystem behavior.
- Keep implementation in `src/` aligned with the standards and validate changes through `tests/` and CI workflows.
