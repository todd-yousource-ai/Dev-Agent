# Enterprise Security Platform

A policy-driven security enforcement platform that provides unified access control, threat detection, and compliance governance across enterprise infrastructure.

## What It Does

This platform enforces security policies across identity, network, data, and application boundaries through a coordinated set of subsystems operating under a zero-trust architecture. It provides real-time threat detection, automated incident response, and continuous compliance verification for enterprise environments. Every request is authenticated, authorized, and audited — no implicit trust between any components.

## Key Subsystems

- **Identity & Access Management (IAM)** — enforces authentication, authorization, and identity lifecycle across all services
- **Policy Engine** — evaluates and enforces security policies at every enforcement point using a centralized rule set
- **Threat Detection** — performs real-time analysis of security events, anomaly detection, and threat correlation
- **Incident Response Orchestrator** — automates containment, remediation, and escalation workflows on detected threats
- **Data Protection** — enforces encryption, classification, and data loss prevention policies on data at rest and in transit
- **Network Security** — manages microsegmentation, firewall rules, and network-level access controls
- **Compliance Engine** — continuously validates infrastructure and operations against regulatory frameworks
- **Audit & Logging** — provides tamper-evident audit trails and centralized log aggregation for all enforcement actions
- **Secret Management** — handles secure storage, rotation, and access control for credentials and cryptographic keys

## Architecture Overview

All subsystems enforce policy through a layered trust model: requests pass through authentication, then policy evaluation, then enforcement — in that strict order. The Policy Engine is the single source of truth; no subsystem bypasses it. Subsystems communicate over authenticated, encrypted channels with mutual TLS, and every inter-service call is logged to the Audit & Logging subsystem for tamper-evident traceability.

## Getting Started

- **Read `CLAUDE.md` first** — it contains the coding instructions, constraints, and conventions that all contributors (human and LLM) must follow.
- **Review the architecture** in `forge-standards/ARCHITECTURE.md` before proposing any design changes.
- **Check `forge-standards/INTERFACES.md`** for wire formats, API contracts, and inter-subsystem protocols.
- **Find the specs** in `forge-docs/` — every subsystem has a corresponding TRD that defines its requirements, enforcement boundaries, and integration points.
- **Run the test suite** with the CI workflow before submitting any pull request.
