# Enterprise Security Platform

A unified security enforcement platform that provides centralized policy management, threat detection, and compliance automation across distributed enterprise environments.

## What It Does

This platform enforces security policies across enterprise infrastructure through a layered architecture of interconnected subsystems. It provides real-time threat detection, automated incident response, identity and access governance, and continuous compliance monitoring — designed for security teams operating at scale across hybrid cloud and on-premises environments.

## Key Subsystems

- **Policy Engine** — centralized policy definition, evaluation, and enforcement across all subsystems
- **Identity & Access Management (IAM)** — authentication, authorization, role-based and attribute-based access control
- **Threat Detection** — real-time anomaly detection, behavioral analysis, and alert correlation
- **Incident Response** — automated playbook execution, case management, and remediation workflows
- **Compliance Manager** — continuous compliance monitoring, evidence collection, and audit reporting
- **Data Protection** — encryption management, data classification, and loss prevention enforcement
- **Network Security** — microsegmentation, traffic analysis, and firewall policy orchestration
- **Audit & Logging** — immutable audit trails, log aggregation, and forensic analysis support
- **Integration Gateway** — secure connectors to SIEM, SOAR, cloud providers, and third-party tools

## Architecture Overview

The platform follows a trust-layered enforcement model where the Policy Engine serves as the root authority — all subsystem actions are evaluated against centralized policies before execution. Subsystems communicate through authenticated, encrypted internal APIs with strict interface contracts, and enforcement follows a defined evaluation order to prevent policy bypass. The architecture ensures that no component can operate outside the bounds of the active policy set.

## Getting Started

- **Read `CLAUDE.md` first** — it contains all LLM coding instructions, naming conventions, and enforcement rules required before writing any code
- **Review `forge-standards/ARCHITECTURE.md`** — understand the trust model and subsystem relationships before making changes
- **Check `forge-standards/INTERFACES.md`** — all API contracts and wire formats are defined here; implementations must conform exactly
- **Browse `forge-docs/`** — the TRDs contain the authoritative requirements; code must trace back to a TRD
- **Run the test suite** — `tests/` covers policy evaluation, access control, and subsystem integration; all tests must pass before PR submission
