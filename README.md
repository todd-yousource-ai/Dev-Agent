# Forge Platform

A security enforcement platform that provides hierarchical trust validation, policy-driven access control, and cryptographic integrity verification across distributed enterprise systems.

## What It Does

Forge enforces security invariants across an enterprise stack through a layered architecture of trust boundaries, policy engines, and integrity checks. It provides real-time policy evaluation, cryptographic attestation of system state, and automated enforcement of security contracts between services. The platform is designed for security teams and platform engineers who need deterministic, auditable control over distributed infrastructure.

## Key Subsystems

- **Trust Boundary Manager** — enforces hierarchical trust zones and validates cross-boundary requests against declared policies
- **Policy Engine** — evaluates access control decisions using a declarative rule language with conflict resolution and audit logging
- **Cryptographic Integrity Service** — provides attestation, signature verification, and tamper detection across system artifacts
- **Identity Resolution** — maps principals across federated identity providers with consistent claim normalization
- **Enforcement Orchestrator** — sequences policy evaluation, trust validation, and integrity checks in a deterministic execution order
- **Audit Ledger** — immutable append-only log of all enforcement decisions, policy changes, and trust boundary modifications
- **Configuration Sync** — distributes policy and trust configuration across nodes with consistency guarantees
- **Health & Observability** — exposes enforcement metrics, decision latency, and subsystem health for operational monitoring

## Architecture Overview

Forge uses a strict enforcement pipeline: every request passes through Trust Boundary validation, then Policy Engine evaluation, then Cryptographic Integrity verification — in that order. No downstream subsystem can override an upstream rejection. The trust model is hierarchical, with zone-level policies inheriting from and constrained by parent zones, ensuring that broader organizational policies cannot be bypassed by local configuration.

## Getting Started

- **Read `CLAUDE.md` first** — it contains the coding constraints, naming conventions, and generation rules that all code in this repo must follow.
- **Review `forge-standards/ARCHITECTURE.md`** — understand the enforcement pipeline order and trust hierarchy before making changes.
- **Check `forge-standards/INTERFACES.md`** — all subsystem communication uses the wire formats and API contracts defined here. Do not invent new ones.
- **Find the spec before you build** — every subsystem has a TRD in `forge-docs/`. Implementation must trace back to a specific TRD requirement.
- **Run the test suite** — `tests/` contains enforcement invariant tests. All must pass before any PR merges.
