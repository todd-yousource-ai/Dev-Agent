# Policy Enforcement Plane
A zero-trust enterprise security platform that makes every access decision explicit, verifiable, and consistently enforced across identity, device, network, and workload boundaries.

## What It Does
Policy Enforcement Plane evaluates and enforces authorization decisions for enterprise users, services, and workloads using a deny-by-default trust model. It protects internal applications, APIs, and service-to-service traffic by combining identity signals, device posture, workload attestation, and policy evaluation into a single enforcement path. The platform is designed for security engineering, platform, and infrastructure teams that need consistent controls across human and machine access.

## Key Subsystems
- Policy Decision Point (PDP) — evaluates access requests against centralized policy and returns permit/deny decisions with obligations.
- Policy Enforcement Point (PEP) — intercepts requests at ingress and service boundaries and enforces PDP decisions before access is granted.
- Identity Federation Gateway — validates user and service identity via OIDC/OAuth 2.0 flows and normalized claims.
- Device Posture Verifier — enforces posture-based access using managed device state and compliance signals.
- Workload Attestation Service — verifies workload identity and runtime provenance before allowing east-west communication.
- Service Mesh Authorization Layer — applies service-to-service policy enforcement and mutual authentication across workloads.
- Audit and Decision Log Pipeline — records policy inputs, decisions, and enforcement outcomes for traceability and review.
- Policy Administration Plane (PAP) — manages versioned policy publication, review, and promotion into enforcement environments.

## Architecture Overview
The platform follows a zero-trust enforcement model: identity is authenticated first, device and workload trust signals are gathered next, and policy is then evaluated before any protected resource is reached. Enforcement happens at dedicated PEPs on ingress and service boundaries, while the PDP remains the centralized authority for decisions and obligations. Trust is continuously derived from signed identity, attestation, and posture evidence rather than network location.

## Getting Started
- Read `CLAUDE.md` first; it defines how code, tests, and changes should be produced in this repo.
- Review `forge-standards/ARCHITECTURE.md` and `forge-standards/INTERFACES.md` before changing any subsystem behavior.
- Use `forge-docs/` as the source of truth for subsystem requirements, trust boundaries, and enforcement flows.
- Keep implementation aligned with the documented architecture rules and interface contracts; do not invent parallel patterns in `src/`.
- Add or update tests in `tests/` for any policy, protocol, or enforcement-path change.
