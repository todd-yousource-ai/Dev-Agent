## 1. Purpose and Scope

This TRD defines the code quality pipeline for the Forge Dev Agent — the complete set of transforms, validations, and iterative fix strategies applied to LLM-generated code before it is committed to GitHub and submitted for CI review.

The pipeline owns: - LLM output sanitization — fence stripping, Unicode normalization, structural validation - Pre-commit validation philosophy — what to block vs. what to let through - The 20-pass iterative fix loop with escalating strategies - Test growth — expanding the test suite with each failed attempt - CI fix loop — autonomous resolution of GitHub Actions failures - Environment parity — ensuring local tests predict CI outcomes

SCOPE NOTE: This TRD does not specify the consensus generation internals (TRD-2), GitHub API operations (TRD-5), or state persistence (TRD-13). It specifies what happens to code after it is generated and before it is merged.

## 2. Design Philosophy

### 2.1 The Copilot Principle

The design of this pipeline is modeled on GitHub Copilot coding agent’s approach to code quality:

Generate → run tests → see failure → fix code → add tests for the failure case → run again → repeat until green → open PR

Key implications: 1. Tests grow with each iteration. The test suite is not static. Every failure teaches the system something new, and that knowledge is encoded as an additional test assertion. 2. LLMs need room to self-correct. Blocking code generation too early (e.g., refusing to proceed on a syntax error) removes the LLM’s context and forces a blind restart. Letting the LLM see the error, the test, and the prior code produces better fixes. 3. CI is the accurate environment. Local macOS tests are necessary but not sufficient. The pipeline must close the gap between the developer’s machine and the clean Ubuntu CI runner.

### 2.2 What We Block Early vs. What We Let Through

Issue | Early block? | Why
Markdown fences in code | Yes | Structural — unfixable by tests
Empty code (< 20 chars) | Yes | Structural — nothing to run
Syntax errors | No | LLM can fix with error context
Logic errors | No | LLM needs test output to diagnose
Ruff lint violations | No | Fix loop handles; CI fix loop catches remainder
Missing imports | No | Auto-install in fix loop
Wrong filename | Partial | Derived from spec title; untitled.py explicitly rejected

The rule: Block only what cannot be recovered by showing the LLM the failure output. Everything else goes to the fix loop.

## 3. LLM Output Sanitization

### 3.1 Overview

All code generated by LLMs passes through _strip_code_fences() before being written to disk or committed to GitHub. This function is implemented identically in all five code generation modules:

build_director.py

consensus.py

failure_handler.py

pr_planner.py

mcp_generator.py

### 3.2 Sanitization Pipeline

Applied in order:

Step 1: Strip markdown fences

LLMs wrap code in markdown fences despite explicit instructions not to. The opening fence may include a language hint (python, swift, go).

lines = code.strip().splitlines()
if lines and lines[0].strip().startswith("```"):
    lines = lines[1:]
if lines and lines[-1].strip() == "```":
    lines = lines[:-1]
result = "\n".join(lines).strip() + "\n"

Step 2: Unicode normalization

LLMs insert typographic Unicode characters that are visually indistinguishable from ASCII but break Python syntax. These must be normalized to their ASCII equivalents.

Unicode character | Code point | ASCII replacement | Why it appears
Left double quote " | U+201C | " | LLM trained on typeset text
Right double quote " | U+201D | " | LLM trained on typeset text
Left single quote ' | U+2018 | ' | LLM trained on typeset text
Right single quote ' | U+2019 | ' | Apostrophes in docstrings
En dash – | U+2013 | - | Comment punctuation
Em dash — | U+2014 | -- | Comment punctuation
Horizontal ellipsis … | U+2026 | ... | Placeholder code
Non-breaking space | U+00A0 | space | Copy-paste from docs

result = result.replace("\u201c", '"').replace("\u201d", '"')
result = result.replace("\u2018", "'").replace("\u2019", "'")
result = result.replace("\u2013", "-").replace("\u2014", "--")
result = result.replace("\u2026", "...").replace("\u00a0", " ")

### 3.3 What is NOT Auto-Fixed

Octal literals (0755, 0644) were initially auto-fixed with a regex transform (0755 → 0o755). This was removed after the regex corrupted reference identifiers inside string literals: ADR-001 became ADR-0o01. Octal syntax errors are handled by the 20-pass fix loop instead, where the LLM can apply the fix correctly with context.

F541 f-strings without placeholders (ruff rule F541) appear frequently when LLMs write multiline string expressions using f-prefixes for consistency. These are informational warnings caught by the CI fix loop; they do not block code generation.

### 3.4 Implementation Requirements

Every _strip_code_fences() function across all five modules must be identical. Any change to sanitization logic must be applied to all five. The function must: - Accept an empty string or None and return it unchanged - Preserve trailing newline on non-empty output - Not modify code that has no fences or unicode characters - Be synchronous (no async)

## 4. Filename Derivation and Path Security

### 4.1 The untitled.py Problem

When spec.impl_files is empty (the PR planner did not specify a filename), code generation must derive a fallback path. The previous fallback was src/untitled.py. This was problematic:

Path security rejected it as a placeholder name

All downstream CI failures showed src/untitled.py regardless of actual PR content

Multiple PRs would overwrite each other’s files

Root cause: The PR planner was generating specs with correct impl_files values in the LLM response, but the spec population code was not reading those fields. Seven fields were populated; impl_files and test_files were silently omitted.

Fix: PRPlanner now explicitly reads impl_files and test_files:

_new_impl_files = spec_data.get("impl_files", [])
if _new_impl_files:
    item.impl_files = [f for f in _new_impl_files if isinstance(f, str) and f.strip()]

### 4.2 Fallback Path Derivation

When spec.impl_files is still empty after the fix (edge case), the path is derived from the spec title using Python module naming:

_title_slug = re.sub(r"[^a-z0-9]+", "_", spec.title.lower())[:40].strip("_")
impl_path = f"src/{_title_slug}.py"
# "Establish repository directory structure" → src/establish_repository_directory_structure.py

### 4.3 Path Security

_sanitize_file_path() validates every model-supplied path before use: - Checks against allowed roots: src/, tests/, scripts/, docs/, ForgeAgent/, ForgeAgentTests/ - Rejects known placeholder names: untitled.py, placeholder.py, temp.py, impl.py, etc. - Falls back to src/impl_generated.py on any violation (never untitled.py)

## 5. Pre-Commit Validation

### 5.1 Structural Check Only

Before committing generated code to GitHub, a lightweight structural check runs. It catches only failures that the fix loop cannot address:

if impl_path.endswith(".py"):
    _is_empty = not exc.impl_code or len(exc.impl_code.strip()) < 20
    _has_fences = exc.impl_code.strip().startswith("```")
    if _is_empty or _has_fences:
        return False  # Regenerate — structural failure

### 5.2 Syntax Errors Pass Through

If ast.parse() raises a SyntaxError, the code is committed anyway and the error is logged. The fix loop handles it:

try:
    _ast.parse(exc.impl_code)
except SyntaxError as _syn:
    logger.warning(f"Pre-commit syntax note: {_syn.msg} line {_syn.lineno} — passing to failure handler")
# Continue to commit and fix loop

Rationale: The fix loop has 20 attempts with escalating strategies including full nuclear regeneration. A syntax error at line 198 with context (the LLM sees the error, the test, and the prior code) produces a better fix than a blind regeneration with no context.

### 5.3 SyntaxError → Fix Loop Routing

When test_runner.write_files() raises a SyntaxError (the file cannot be written), the error is routed to the fix loop rather than skipping the PR:

try:
    impl_local, test_local = self._test_runner.write_files(...)
except SyntaxError as _syn:
    # Write broken file anyway so fix loop has context
    from pathlib import Path as _Path2
    _td = tempfile.mkdtemp()
    impl_local = _Path2(_td) / _Path2(impl_path).name
    test_local = _Path2(_td) / f"test_{_Path2(impl_path).name}"
    impl_local.write_text(exc.impl_code)
    test_local.write_text(exc.test_code or "# placeholder")
    # Fall through to 20-pass fix loop

## 6. The 20-Pass Iterative Fix Loop

### 6.1 Overview

MAX_TEST_RETRIES = 20 — hard cap matching GitHub Copilot coding agent’s iteration depth.

The loop runs locally using pytest. All failures are visible — syntax errors, logic errors, and lint violations. The failure handler uses dual-LLM consensus: both Claude and ChatGPT diagnose and fix independently, Claude arbitrates.

### 6.2 Escalating Strategy

Attempts | Strategy | Description
1 | system_diagnosis | General diagnosis — root cause category, evidence, proposed fix
2 | test_currency_audit | Are tests stale? Do they reflect the current interface?
3 | full_branch_audit | Read up to 25 branch files for full repository context
4–7 | test_driven_fix | Extract exact failing assertions, inject as hard constraint: “The test is the spec”
8–10 | nuclear_regenerate | Discard current impl entirely, regenerate from test + spec
11–13 | test_driven_fix | Resume test-driven with now-grown test suite
14–15 | nuclear_regenerate | Nuclear with grown suite
16–17 | test_driven_fix | Final test-driven passes
18–20 | nuclear_regenerate | Last resort

### 6.3 Test-Driven Fix Strategy

From attempt 4 onward, the failing test assertions are extracted and injected as a hard constraint into the fix prompt:

failing_tests = [
    line.strip() for line in (result.stdout + result.stderr).splitlines()
    if "FAILED" in line or "AssertionError" in line or "assert " in line.lower()
][:10]

test_constraint = (
    "\n\nCRITICAL: These specific tests are failing and MUST pass:\n"
    + "\n".join(f"  {t}" for t in failing_tests)
    + "\n\nRead the test code carefully. Generate implementation that satisfies "
    "each failing assertion exactly. The test is the spec — do not change the tests."
)

### 6.4 Nuclear Regenerate Strategy

On nuclear attempts, the current implementation is discarded entirely:

if strategy == "nuclear_regenerate":
    current_impl = ""
    print("  ⚠  Nuclear regenerate — discarding current impl, rebuilding from test spec")

The LLM receives only the spec, the test file, and the failure output — no prior broken code to anchor on.

### 6.5 Test Growth

From attempt 4 onward, after every fix attempt both Claude and ChatGPT independently expand the test suite with new assertions covering the specific failure case. The longer test suite wins:

if attempt >= 4 and fix.get("fixed_impl"):
    growth_prompt = (
        f"The following test failure just occurred:\n{failure_output[:800]}\n\n"
        f"Current test file:\n{current_tests[:2000]}\n\n"
        "Add new test assertions that specifically cover this failure case. "
        "Do NOT remove existing tests. Return the complete expanded test file. "
        "Return ONLY the Python test code, no markdown fences."
    )
    _grown_claude, _grown_openai = await asyncio.gather(
        self._claude_text(_SYSTEM_FIX, growth_prompt),
        self._openai_text(_SYSTEM_FIX, growth_prompt),
    )
    _grown = _grown_claude if len(_grown_claude) >= len(_grown_openai) else _grown_openai
    _grown = _strip_code_fences(_grown)
    if len(_grown) > len(current_tests):
        current_tests = _grown
        test_local.write_text(current_tests)
        print(f"  ✚  Test suite expanded ({len(current_tests.splitlines())} lines)")

Observable: ✚ Test suite expanded (N lines) in the build log.

### 6.6 Consensus Fix Architecture

Each fix attempt: 1. Both Claude and ChatGPT diagnose independently 2. If both agree on root_cause_category → use Claude’s diagnosis (more detailed) 3. If they disagree → Claude arbitrates between the two diagnoses 4. Both models produce independent code fixes 5. Claude arbitrates: picks the fix that addresses more diagnosed issues 6. If arbitration fails → fall back to Claude’s fix

## 7. CI Fix Loop

### 7.1 Why CI is the Accurate Environment

GitHub Actions CI (Ubuntu Linux) is the accurate test environment. Local macOS tests are necessary but not sufficient. The gap between environments is systematic and predictable:

Gap | Local macOS | CI Ubuntu | Impact
Installed packages | Pre-installed globally | requirements.txt only | Import errors pass locally, fail CI
Filesystem case sensitivity | Case-insensitive | Case-sensitive | Path resolution differs
Environment variables | .env loaded | None | Missing config causes runtime errors
Ruff version | Local installed version | CI pinned version | Lint rules may differ
Python minor version | 3.14.0 | Pinned in CI workflow | Minor syntax/behavior differences

### 7.2 CI Fix Loop Protocol

When CI fails after local tests pass, the CI fix loop runs autonomously:

CI fails
  → get_ci_failure_logs() fetches GitHub Actions annotations
  → run_fix_loop() with CI logs as prd_context (5 attempts, Ubuntu-aware)
  → commit fixed code
  → wait for CI
  → if still failing: repeat (max 3 CI cycles)
  → if all 3 cycles fail: mark PR as draft, move to next PR

CI cycle limit: CI_FIX_ATTEMPTS = 3 — prevents runaway cost. Each CI cycle takes 3–5 minutes.

### 7.3 CI Log Fetching

GitHub Actions annotations are fetched using the check run API:

def get_ci_failure_logs(self, branch: str, max_chars: int = 3000) -> str:
    repo = self._github._get_repo()
    commit = repo.get_branch(branch).commit
    checks = list(commit.get_check_runs())
    failed_checks = [c for c in checks if c.conclusion not in ("success", "neutral", "skipped", None)]
    
    log_lines = []
    for check in failed_checks[:2]:
        log_lines.append(f"=== CI Failure: {check.name} ===")
        annotations = list(check.get_annotations())
        for ann in annotations[:20]:
            log_lines.append(f"{ann.path}:{ann.start_line}: {ann.annotation_level}: {ann.message}")
    
    return "\n".join(log_lines)[:max_chars]

### 7.4 CI Environment Context Injection

When the failure handler is called with CI logs, it automatically detects the CI context and injects Ubuntu environment awareness:

IMPORTANT: This failure came from GitHub Actions CI (Ubuntu Linux), not local macOS.
Common causes:
- Missing package in requirements.txt (installed locally but not listed)
- macOS vs Linux path differences (case-sensitive filesystem on Linux)
- Missing environment variables (CI has no .env file)
- Python 2 style syntax (octal literals 0755, print statements)
- Ruff version differences (CI may be stricter)
Fix must work on clean Ubuntu Linux with only requirements.txt packages.

### 7.5 Automatic requirements.txt Update

When a missing package import error appears in CI logs:

if failure_type == "import_error" and _is_ci_fix:
    pkg = _extract_missing_package(error_lines)
    # Install locally
    subprocess.run([sys.executable, "-m", "pip", "install", "--quiet", pkg])
    # Add to requirements.txt so CI has it
    req_path = Path(__file__).parent.parent / "requirements.txt"
    if pkg.lower() not in req_path.read_text().lower():
        req_path.open("a").write(f"\n{pkg}\n")

### 7.6 Final Ruff Parity Check

Before the final commit to GitHub (after all local fix attempts), a ruff check runs locally to match CI’s lint check. This is informational — it logs warnings but does not block the commit after 20+ fix attempts have already run.

## 8. Cost Controls

### 8.1 Token Budget per PR

Phase | Attempts | Token estimate | Cost estimate
Code generation | 1 | ~12,000 | $0.18
Fix loop (local, clean) | 0 | $0
Fix loop (local, 5 attempts) | 5 | ~60,000 | $0.90
Fix loop (local, 20 attempts) | 20 | ~240,000 | $3.60
CI fix loop (1 cycle, 5 attempts) | 5 | ~60,000 | $0.90
CI fix loop (3 cycles) | 15 | ~180,000 | $2.70

Expected per PR: $0.18 (best case, clean) to $7.38 (worst case, 20 local + 3 CI cycles)

Budget alarm: If a single PR consumes more than $10 in fix attempts, it indicates a fundamental spec/test mismatch that manual intervention is more efficient than continued automation.

### 8.2 Hard Caps

Cap | Value | Rationale
MAX_TEST_RETRIES | 20 | Matches Copilot agent depth; 21+ attempts indicate spec/impl fundamental mismatch
CI_FIX_ATTEMPTS | 3 | Each CI cycle is 3–5 min + token cost; 3 cycles covers all systematic environment gaps
Fix loop per CI attempt | 5 | Focused on CI-specific issue; full 20-pass loop would be overkill

## 9. Testing Requirements

### 9.1 Unit Tests

Module | Test cases
_strip_code_fences() | Empty input; no fences; opening fence only; closing fence only; both fences; python hint; swift hint; curly quotes; em dash; non-breaking space
Filename derivation | Empty impl_files; title with special chars; title truncation at 40 chars; untitled rejected; impl_generated fallback
Path security | All allowed roots pass; traversal rejected; placeholder names rejected
Fix loop strategy rotation | Attempt 1–3 correct strategies; attempt 4 is test_driven_fix; attempt 8 is nuclear; attempt 20 is nuclear
Test growth | Not triggered before attempt 4; triggered at attempt 4; longer suite wins; strip_fences applied to grown tests
CI log fetching | Failed checks returned; annotations formatted; max_chars respected
requirements.txt update | Package added if absent; not added if present; file created if missing

### 9.2 Integration Tests

Full PR generation → local test failure → fix loop → local pass → CI pass (happy path)

Local pass → CI fail → CI fix loop → CI pass

Local pass → CI fail (missing package) → auto-add to requirements.txt → CI pass

20 attempts exhausted → PR marked as draft → build continues to next PR
