# Product Foundation, Repository Bootstrap, and Cross-TRD Contract Baseline

**PRD ID:** PRD-001
**Status:** Draft
**Author:** Forge Dev Agent (AI Consensus)

---

## Overview

PRD-001 establishes the foundational infrastructure, naming conventions, versioning contracts, directory layout, configuration model, and cross-TRD conflict resolution hierarchy required before any feature work begins on the Consensus Dev Agent. This PRD produces no user-visible functionality. It produces the structural preconditions that every subsequent PRD depends on: a canonical product identity, a bootstrapped repository skeleton for both the Swift shell and Python backend, interface contracts codified in machine-readable form, a semantic versioning scheme spanning both processes and all CI artifacts, naming conventions for branches, files, artifacts, and state, a shared configuration model, and a single source-of-truth hierarchy that resolves all ambiguities across the 16 TRDs.

Without this baseline committed to the repository, parallel workstreams will independently invent incompatible conventions for directory structure, version strings, configuration keys, branch naming, file placement, error type naming, and cross-TRD precedence. Those incompatibilities compound across every subsequent PRD and produce integration failures that cost more to resolve than the original feature work.

This PRD is dependency-free. Every other PRD in the system depends on it.

---

## Problem Statement

The Consensus Dev Agent is specified across 16 TRDs (TRD-1 through TRD-16), each authored independently. These TRDs reference shared concepts — version strings, directory paths, configuration keys, naming conventions, error hierarchies, XPC message schemas — but define them inconsistently or leave them implicit. Specific problems:

1. **No single version source of truth.** TRD-1 references `CFBundleShortVersionString` in `Info.plist`. TRD-12 references `VERSION` file and `AGENT_VERSION` constant in `src/version.py`. TRD-3 references `pyproject.toml`. AGENTS.md references a `VERSION` file that must match `pyproject.toml`. No document defines the authoritative relationship between these four version representations or how they are kept in sync across Swift and Python processes during the version handshake (TRD-12 §4).

2. **No conflict resolution hierarchy is codified.** CLAUDE.md states "TRD-11 is authoritative over all other TRDs on security questions." AGENTS.md states "TRD-11 governs all components." Neither document defines a complete precedence order for non-security conflicts (e.g., when TRD-1 and TRD-12 disagree on startup timeout values, or when TRD-3 and TRD-14 define overlapping code quality gates).

3. **Directory layout is described but not enforced.** AGENTS.md shows a directory tree. README.md shows a different directory tree. Neither matches what exists at commit zero. Placeholder directories (`forge-standards/`, `ForgeAgent/`, `ForgeAgentTests/`) are referenced by multiple TRDs but do not exist in the repository.

4. **Naming conventions are implicit.** Branch naming is defined in AGENTS.md (`forge-agent/build/{engineer_id}/{subsystem_slug}/pr-{N:03d}-{title_slug}`). File naming for PR plan artifacts is defined in TRD-4 (`prds/{subsystem_slug}/{prd_id}-pr-plan.json`). Configuration key naming follows no declared convention. Error type naming differs between TRD-1 (Swift enums), TRD-5 (Python exception hierarchy), and TRD-10 (Python enums). No document defines the canonical naming rules that all TRDs must follow.

5. **Configuration model is fragmented.** TRD-1 §3.4.3 defines `SettingsStore` with `@AppStorage` keys. TRD-10 §4.1 defines chunking parameters as module-level constants. TRD-12 §3.1 defines `CREDENTIAL_TIMEOUT_SEC` as a constant. TRD-4 §8.1 defines `HEARTBEAT_INTERVAL_SEC`. No document defines whether these are runtime-configurable, compile-time constants, or operator-adjustable via Settings. No document defines how Python backend configuration is structured or validated.

6. **No bootstrap assets exist.** The repository at commit zero contains no `VERSION` file, no `pyproject.toml`, no `forge-standards/` directory, no `ARCHITECTURE.md`, no `INTERFACES.md`, no `CONVENTIONS.md`, no `DECISIONS.md`, no Swift project skeleton, no Python package structure, and no `.gitkeep` markers for required empty directories. Every TRD assumes these exist.

7. **Ready handshake compatibility is underspecified.** TRD-12 §5 defines compatibility rules (major must match, backend minor >= shell minor) but does not define how `MIN_SWIFT_VERSION` is determined at build time, how version constants are injected into both processes, or how CI validates version consistency across the two processes.

## Goals and Non-Goals

### Goals

G-1. **Establish canonical product identity.** Define "Forge" as the primary product name, "Dev Agent" as the subheader, "Consensus Dev Agent" as the full technical name, and `ai.yousource.forgeagent` as the bundle identifier. All generated artifacts, commit messages, branch names, and documentation must use these names consistently.

G-2. **Create the repository skeleton.** Commit the complete directory structure referenced by all 16 TRDs, including placeholder directories with `.gitkeep`, the Python package structure (`src/` with `__init__.py`, `tests/` with `conftest.py`), the Swift project directory (`ForgeAgent/`, `ForgeAgentTests/`), `forge-standards/`, `forge-docs/`, and `.github/workflows/`.

G-3. **Define and commit the VERSION contract.** Create the `VERSION` file as the single source of truth. Define the exact mechanism by which `pyproject.toml`, `Info.plist` (`CFBundleShortVersionString`), `src/version.py` (`AGENT_VERSION`), and the XPC ready message `agent_version` field derive from it. Define the CI test that enforces consistency.

G-4. **Codify the TRD conflict resolution hierarchy.** Produce `forge-standards/DECISIONS.md` with the complete precedence order: TRD-11 overrides all TRDs on security matters; the owning TRD overrides all other TRDs for its owned domain; `forge-standards/` documents override implicit TRD assumptions; later TRD versions override earlier versions of the same TRD.

G-5. **Define and commit all naming conventions.** Produce `forge-standards/CONVENTIONS.md` covering: branch names, file names (Python modules, Swift files, test files, configuration files), artifact names (CI artifacts, build outputs, dSYMs), state file names (thread state, ledger, manifests), configuration key names (Python and Swift), error type names, XPC message type names, and PR/PRD identifier formats.

G-6. **Define and commit the shared configuration model.** Produce `forge-standards/ARCHITECTURE.md` with the configuration taxonomy: compile-time constants (never change at runtime), operator-adjustable settings (UserDefaults in Swift, `config.py` in Python), runtime-derived values (computed from environment), and secret values (Keychain only, delivered via XPC). Produce `src/config.py` with the Python `AgentConfig` dataclass. Define the `VERSION` file, capability flags, and the ready handshake compatibility validation.

G-7. **Define and commit interface contracts.** Produce `forge-standards/INTERFACES.md` with the canonical XPC message type table, the `ready` message schema, the `credentials` message schema, the Python backend public API surface (method signatures for `ConsensusEngine`, `GitHubTool`, `DocumentStore`, `BuildLedger`), and the Swift module public API surface (type signatures for `AuthManager`, `KeychainManager`, `XPCChannel`, `BackendProcess`).

G-8. **Resolve all identified cross-TRD ambiguities.** Document each resolution in `forge-standards/DECISIONS.md` with: the conflicting TRDs, the ambiguity, the resolution, the rationale, and the owning TRD going forward.

G-9. **Produce AGENTS.md context file at repository root.** Update the AGENTS.md to reflect the committed directory structure, version contract, and naming conventions — ensuring it is accurate at commit zero, not aspirational.

G-10. **Produce architecture context file.** Commit `forge-standards/ARCHITECTURE.md` (or update the existing `forge_architecture_context` asset) to reflect the canonical architecture, startup sequence, and interface contracts as defined in this PRD.

### Non-Goals

NG-1. **Implement any functional component.** This PRD produces no working Swift code, no working Python code beyond `config.py` and `version.py` stubs, and no working CI pipeline. It produces the skeleton and contracts that functional PRDs build into.

NG-2. **Create the Xcode project file.** The `.xcodeproj` or Swift Package Manager `Package.swift` is created by a subsequent PRD that implements TRD-1. This PRD creates the directory structure and placeholder files that the Xcode project will reference.

NG-3. **Implement the build pipeline.** Pipeline stages (TRD-3), consensus engine (TRD-2), GitHub integration (TRD-5), and all other functional subsystems are implemented in their own PRDs.

NG-4. **Define UI components.** All SwiftUI views, design tokens, and visual specifications are deferred to PRDs implementing TRD-8.

NG-5. **Set up the Mac CI runner.** Runner installation, certificate setup, and macOS workflow are deferred to a PRD implementing TRD-9.

NG-6. **Resolve open questions in individual TRDs.** Each TRD has OQ-* items. This PRD resolves only cross-TRD ambiguities. Intra-TRD open questions are resolved by the PRD that implements that TRD.

## User Stories

US-1. **As a developer agent beginning work on any TRD,** I need to know the exact directory path where my files must be placed, so that I do not invent a path that conflicts with another TRD's expected structure.

US-2. **As a developer agent generating a version string,** I need a single authoritative source (`VERSION` file) and a defined derivation mechanism for every other version representation, so that version consistency is enforced by CI, not by convention.

US-3. **As a developer agent encountering a conflict between two TRDs,** I need a documented precedence hierarchy, so that I resolve the conflict deterministically without human intervention.

US-4. **As a developer agent naming a branch, file, configuration key, or error type,** I need a documented naming convention, so that all names across the repository follow a single pattern.

US-5. **As a developer agent implementing `config.py`,** I need a defined configuration taxonomy that classifies every parameter as compile-time, operator-adjustable, runtime-derived, or secret, so that I do not expose secrets as configuration or make compile-time constants runtime-adjustable.

US-6. **As a developer agent implementing the Python backend entry point,** I need `src/config.py` with the `AgentConfig` dataclass and `src/version.py` with `_read_version()`, so that I can import configuration and version from the canonical location.

US-7. **As a developer agent implementing the XPC handshake,** I need the interface contract for the `ready` message schema, including `agent_version`, `min_swift_version`, `capabilities`, and `doc_store_status`, defined in a single machine-readable location, so that Swift and Python implementations match exactly.

US-8. **As a CI pipeline validating a PR,** I need a test that verifies `VERSION`, `pyproject.toml`, and `src/version.py` all agree, so that version drift is caught before merge.

US-9. **As the repository at commit zero,** I need all placeholder directories to exist with `.gitkeep` files, so that `git clone` produces the complete directory structure expected by all TRDs.

US-10. **As a developer agent implementing any module,** I need `forge-standards/INTERFACES.md` with the public API signatures of all cross-module interfaces, so that I implement against the correct contract without reading multiple TRDs.

## Functional Requirements

### FR-1: VERSION File and Derivation Contract

FR-1.1. A file named `VERSION` shall exist at the repository root containing a single line: a semantic version string in the format `MAJOR.MINOR.PATCH` (e.g., `1.0.0`). No trailing newline beyond the final character.

FR-1.2. `pyproject.toml` shall exist at the repository root. The `[project]` section shall contain a `version` field whose value is identical to the content of `VERSION`.

FR-1.3. `src/version.py` shall exist and define:
- `AGENT_VERSION: str` — hardcoded fallback, matching `VERSION` at commit time.
- `MIN_SWIFT_VERSION: str` — the minimum Swift shell version this backend requires, initially `"1.0.0"`.
- `def _read_version() -> str` — reads `VERSION` file at runtime, falls back to `AGENT_VERSION`.

FR-1.4. The Swift shell shall read its version from `Info.plist` key `CFBundleShortVersionString` and a custom key `ForgeAgentShellVersion`. Both shall be set at build time from the `VERSION` file via a build phase script.

FR-1.5. The initial version shall be `1.0.0`.

FR-1.6. A Python test `tests/test_version.py::test_version_consistency` shall verify that `VERSION`, `pyproject.toml` version, and `src/version.py` `AGENT_VERSION` all match. This test shall be included in the unit test suite and run on every CI invocation.

### FR-2: Directory Structure

FR-2.1. The following directories shall exist at commit zero, each containing a `.gitkeep` file if otherwise empty:

```
forge-docs/
forge-standards/
src/
tests/
ForgeAgent/
ForgeAgentTests/
.github/workflows/
scripts/
```

FR-2.2. `src/` shall contain `__init__.py` (empty).

FR-2.3. `tests/` shall contain `__init__.py` (empty) and `conftest.py` (with `sys.path` insertion for `src/`).

FR-2.4. `forge-standards/` shall contain the four standards documents: `ARCHITECTURE.md`, `INTERFACES.md`, `DECISIONS.md`, `CONVENTIONS.md`.

FR-2.5. `forge-docs/` shall be reserved for TRD and PRD documents. No code or configuration shall be placed here.

FR-2.6. `.github/workflows/` shall contain placeholder workflow files (`forge-ci.yml` and `forge-ci-macos.yml`) with comments indicating they are stubs to be replaced by functional PRDs.

### FR-3: Conflict Resolution Hierarchy

FR-3.1. `forge-standards/DECISIONS.md` shall define the following precedence order for resolving conflicts between documents:

1. **TRD-11 (Security Threat Model)** — overrides all other TRDs on any security-related requirement. If a non-security TRD defines a behavior that conflicts with a SEC-* control in TRD-11, TRD-11 wins unconditionally.
2. **Owning TRD** — for its owned domain, the owning TRD overrides all other TRDs. The ownership map is the TRD Index in README.md.
3. **forge-standards/ documents** — override implicit assumptions in TRDs. If a TRD assumes a convention not documented in `forge-standards/`, the `forge-standards/` document is authoritative once committed.
4. **Later TRD version** — a v2.0 of a TRD overrides v1.0 of the same TRD. Version is stated in the TRD header.
5. **PRD-001 resolutions** — specific ambiguity resolutions documented in `DECISIONS.md` override all other sources for the specific ambiguity addressed.

FR-3.2. `DECISIONS.md` shall document at minimum the following resolved ambiguities:

- **DEC-001: Startup credential timeout.** TRD-12 §3.1 specifies `CREDENTIAL_TIMEOUT_SEC = 30`. TRD-1 §7.1 references a startup sequence but does not specify the timeout. Resolution: TRD-12 owns the Python backend startup; 30 seconds is authoritative. TRD-1 shall use this value when validating backend startup.

- **DEC-002: XPC message type enumeration.** TRD-1 §6.4 defines Swift→Python and Python→Swift message types. TRD-12 §4.1 defines the `ready` message schema. The canonical enumeration of all message types shall be in `forge-standards/INTERFACES.md`, derived from TRD-1 §6.4, with the `ready` message schema from TRD-12 §4.1 incorporated.

- **DEC-003: Configuration key naming.** TRD-1 §3.4.3 uses `@AppStorage("biometric_timeout_sec")`. TRD-10 §4.1 uses `MAX_CHUNK_TOKENS = 512`. TRD-4 §8.1 uses `HEARTBEAT_INTERVAL_SEC = 60`. Resolution: Python backend constants use `UPPER_SNAKE_CASE`. Swift `@AppStorage` keys use `lower_snake_case`. Both are documented in `CONVENTIONS.md`.

- **DEC-004: Code quality gate overlap.** TRD-3 §6 defines a 3-pass review (correctness, performance, security). TRD-14 §6 defines a 20-pass iterative fix loop with escalating strategies. TRD-6 defines a 5-lens holistic review. Resolution: TRD-3 owns the per-PR pipeline review. TRD-14 owns the iterative fix loop within that pipeline. TRD-6 owns the standalone holistic review workflow. They do not conflict — they are three distinct processes with separate triggers.

- **DEC-005: Branch naming authority.** AGENTS.md defines `forge-agent/build/{engineer_id}/{subsystem_slug}/pr-{N:03d}-{title_slug}`. TRD-5 §6.1 defines `forge-agent/build/{engineer_id}/{subsystem_slug}-pr{NNN}-{title_slug}`. Resolution: TRD-5 owns GitHub integration including branch naming. TRD-5 §6.1 format is authoritative. AGENTS.md shall be updated to match.

- **DEC-006: Error type naming.** TRD-1 Appendix A defines Swift error enums (`AuthError`, `KeychainError`, `XPCError`, `ProcessError`, `DocImportError`). TRD-5 §14 defines Python error classes (`GitHubToolError` hierarchy). TRD-10 defines `DocumentParseError`. Resolution: Swift errors use `{Module}Error` enum naming. Python errors use `{Component}Error` class naming with inheritance from a base `ForgeAgentError`. Documented in `CONVENTIONS.md`.

- **DEC-007: State file location.** TRD-3 references `ThreadStateStore` with checkpointing. TRD-13 §6.2 specifies `workspace/todd-gould/state/threads/`. TRD-15 §11 specifies full paths. Resolution: Thread state is stored at `{FORGE_WORKSPACE}/{engineer_id}/state/threads/{subsystem_slug}.json`. `FORGE_WORKSPACE` defaults to `~/Library/Application Support/ForgeAgent/workspace/`. TRD-13 owns the state persistence schema.

- **DEC-008: Default embedding model.** TRD-10 §5.1 specifies `all-mpnet-base-v2` as default. CLAUDE.md §Document Store also references this. Resolution: TRD-10 owns the DocumentStore; `all-mpnet-base-v2` is the canonical default. The model ID is stored in `AgentConfig` and overridable in Settings.

### FR-4: Naming Conventions

FR-4.1. `forge-standards/CONVENTIONS.md` shall define conventions for:

- **Branch names:** `forge-agent/build/{engineer_id}/{subsystem_slug}-pr{NNN}-{title_slug}` per TRD-5 §6.1. PRD storage branches: `forge-agent/build/{engineer_id}/{subsystem_slug}/prds`. Review branches: `forge-review/{engineer_id}/{branch_slug}-{session_short_id}` per TRD-6 §11.1.

- **Python file names:** `lower_snake_case.py`. Module names match file names. Test files: `test_{module_name}.py`.

- **Swift file names:** `PascalCase.swift`. Type name matches file name. Test files: `{TypeName}Tests.swift`.

- **Configuration keys (Python):** Module-level constants in `UPPER_SNAKE_CASE`. Runtime-configurable values as fields on `AgentConfig` dataclass in `lower_snake_case`.

- **Configuration keys (Swift):** `@AppStorage` keys in `lower_snake_case` (e.g., `"biometric_timeout_sec"`).

- **Error types (Python):** Classes inheriting from `ForgeAgentError`. Named `{Component}Error` (e.g., `GitHubToolError`, `DocumentParseError`, `LedgerConflictError`).

- **Error types (Swift):** Enums conforming to `LocalizedError`. Named `{Module}Error` (e.g., `AuthError`, `KeychainError`).

- **XPC message types:** `lower_snake_case` strings (e.g., `"credentials"`, `"gate_response"`, `"build_card"`, `"shutdown_ack"`).

- **PRD identifiers:** `PRD-{NNN}` (zero-padded three digits).

- **PR identifiers (within a build):** `PR{NNN}` in titles, `pr_num` integer in data structures.

- **Artifact names:** `forge-agent-{signed|unsigned}-{run_id}`, `ForgeAgent-{version}.dmg`, `test-results-{run_id}`.

- **State files:** `{subsystem_slug}.json` in the thread state directory.

- **Commit messages:** `forge-agent[{engineer_id}]: {description}` for code commits. `forge-ledger[{engineer_id}]: {description}` for ledger writes. `forge-review[{engineer_id}]: {description}` for review commits.

### FR-5: Shared Configuration Model

FR-5.1. `src/config.py` shall define an `AgentConfig` dataclass with the following field categories:

- **Compile-time constants** (class-level, `UPPER_SNAKE_CASE`): `MAX_CHUNK_TOKENS = 512`, `MIN_CHUNK_TOKENS = 64`, `OVERLAP_TOKENS = 50`, `CONTEXT_BUDGET_CHARS = 24_000`, `CREDENTIAL_TIMEOUT_SEC = 30`, `HEARTBEAT_INTERVAL_SEC = 60`, `STALE_THRESHOLD_SEC = 600`, `MAX_TEST_RETRIES = 20`, `CI_FIX_ATTEMPTS = 3`, `MAX_MESSAGE_BYTES = 16_777_216`, `XPC_RATE_LIMIT_PER_SEC = 100`.

- **Operator-adjustable fields** (instance fields, `lower_snake_case`): `default_repo_owner: str`, `default_repo_name: str`, `pr_batch_size: int = 5`, `cost_warn_threshold: float = 0.50`, `cost_stop_threshold: float = 2.00`, `embedding_model: str = "all-mpnet-base-v2"`, `default_branch: str = "main"`.

- **Runtime-derived fields** (computed, not stored): `workspace_path: Path` (from `FORGE_WORKSPACE` env or default), `log_path: Path` (from `FORGE_LOG_DIR` env or default), `socket_path: str` (from `FORGE_XPC_SOCKET` env), `xpc_nonce: str` (from `FORGE_XPC_NONCE` env).

- **Secret fields:** Not stored in `AgentConfig`. Delivered via XPC `credentials` message and held in memory by the receiving component (`GitHubTool`, `ConsensusEngine`). `AgentConfig` stores only `has_anthropic_key: bool`, `has_openai_key: bool`, `has_github_token: bool` — presence flags, not values.

FR-5.2. `AgentConfig` shall provide a `from_env()` classmethod that populates runtime-derived fields from environment variables and validates that required variables are present.

FR-5.3. `AgentConfig` shall provide a `validate()` method that raises `ConfigurationError` if required fields are missing or invalid (e.g., `pr_batch_size < 1`, `cost_warn_threshold >= cost_stop_threshold`).

### FR-6: Interface Contracts

FR-6.1. `forge-standards/INTERFACES.md` shall define the following contracts:

- **XPC message type table:** Complete enumeration of all message types (Swift→Python and Python→Swift) with required payload fields and types, derived from TRD-1 §6.4 and TRD-12 §4.1.

- **`ready` message schema:** Exact JSON schema including `type`, `id`, `session_id`, `timestamp`, `agent_version`, `min_swift_version`, `capabilities` (array of strings), `doc_store_status` (`"loading"` | `"ready"` | `"unavailable"`), `python_version`, `platform`.

- **Capability flag enumeration:** The complete list of capability strings that may appear in the `ready` message `capabilities` array: `"consensus_engine"`, `"build_pipeline"`, `"github_integration"`, `"multi_agent_ledger"`, `"holistic_review"`, `"trd_workflow"`, `"doc_store"`.

- **Python backend public API surface:** Method signatures (name, parameters with types, return type, raised exceptions) for: `ConsensusEngine.run()`, `GitHubTool` (all 24 methods per TRD-5 Appendix A), `DocumentStore` (all methods per TRD-10 §12), `BuildLedger` (all methods per TRD-4 §14).

- **Swift module public API surface:** Type signatures for: `AuthManager.authenticate()`, `KeychainManager.store()/.read()/.exists()/.delete()`, `XPCChannel.accept()/.send()/.messages`, `BackendProcess.launch()/.terminate()`.

## Non-Functional Requirements (Security, Performance, Compliance)

### Security

NFR-SEC-1. No credential, API key, token, or secret shall appear in any file committed by this PRD. The `VERSION` file, `pyproject.toml`, `config.py`, and all standards documents contain no sensitive material.

NFR-SEC-2. `src/config.py` shall not store, accept, or expose secret values. Secret presence is tracked via boolean flags only (`has_anthropic_key`, `has_openai_key`, `has_github_token`). The `AgentConfig` docstring shall include a prominent warning: "Secrets are never stored in AgentConfig. They are delivered via XPC and held in memory by the consuming component."

NFR-SEC-3. `.gitignore` shall include `.env` to prevent accidental commit of environment files containing secrets.

NFR-SEC-4. `forge-standards/DECISIONS.md` shall include DEC-SEC-001: "TRD-11 is the authoritative security document. Any conflict between TRD-11 and any other TRD is resolved in favor of TRD-11. No exception. No override mechanism."

### Performance

NFR-PERF-1. `AgentConfig.from_env()` shall complete in less than 10 milliseconds. It reads environment variables and validates — no I/O, no network.

NFR-PERF-2. `AgentConfig.validate()` shall complete in less than 1 millisecond. It checks field values — pure computation.

NFR-PERF-3. `_read_version()` shall complete in less than 5 milliseconds. It reads a single small file.

### Compliance

NFR-COMP-1. All committed files shall use UTF-8 encoding with LF line endings.

NFR-COMP-2. All Python files shall pass `ruff check` with the configuration defined in `pyproject.toml`.

NFR-COMP-3. All Python files shall pass `mypy` with the configuration defined in `pyproject.toml`.

## System Architecture Notes

This PRD does not introduce new architectural components. It documents and codifies the architecture defined across the 16 TRDs. Key architectural assertions that this PRD makes authoritative:

1. **Two-process model is mandatory.** Swift shell and Python backend. No third process. No merging of processes.

2. **XPC is the only inter-process communication mechanism.** Unix domain socket, line-delimited JSON, nonce-authenticated. No shared files, no shared memory, no named pipes, no HTTP.

3. **Python backend is the sole owner of all LLM API calls.** Swift never calls Anthropic, OpenAI, or any LLM API. This is a security boundary, not an implementation convenience.

4. **Keychain is the sole credential store.** No credentials in UserDefaults, plist, environment variables persisted to disk, or source code. Python receives credentials via XPC only.

5. **`path_security.validate_write_path()` is called before every file write.** This applies to GitHub commits (via `GitHubTool.commit_file()`), local file writes, and any other file system operation.

6. **Gates never auto-approve.** Every operator gate waits indefinitely. There is no timeout, no auto-merge, no implicit approval.
