# Code Conventions

## File and Directory Naming

1. **Place required repository-root files at the repo root with no directory prefix.**  
   The following files must be committed exactly by basename at the repository root:
   - `README.md`
   - `CODEOWNERS`
   - `LICENSE`
   - `.gitignore`
   - `.editorconfig`
   - `VERSION`
   - `Makefile`
   - `pyproject.toml`

2. **Do not relocate or namespace repository-root required files.**  
   Never place required root files under folders such as `docs/`, `.github/`, `config/`, or `meta/` as substitutes for the canonical root file.

3. **Use the declared CI workflow filenames exactly.**  
   CI workflow files must be named:
   - `crafted-ci.yml` for Ubuntu workflows
   - `crafted-ci-macos.yml` for macOS Swift workflows

4. **Keep Python application modules under `src/`.**  
   Python backend source files for CraftedApp must live under `src/`, following the established examples:
   - `src/consensus.py`
   - `src/build_director.py`

5. **Name Python modules in lowercase snake_case.**  
   Use filenames like `build_director.py`, `path_security.py`, and `consensus.py`. Do not use hyphens, spaces, or CamelCase in Python filenames.

6. **Match module filenames to their primary responsibility.**  
   Use singular, responsibility-based names for modules. Example mappings from the TRDs:
   - `src/consensus.py` contains `ConsensusEngine`
   - `src/build_director.py` contains `BuildPipeline`

7. **Use stable subsystem slugs in branch names.**  
   When creating branches for CraftedApp work, use the mandatory format exactly:  
   `forge-agent/build/{engineer_id}/{subsystem_slug}/pr-{N:03d}-{title_slug}`

8. **Zero-pad PR numbers to three digits in branch names.**  
   Valid examples:
   - `pr-001-add-auth-shell`
   - `pr-042-fix-stream-controls`

9. **Use lowercase kebab-case for `subsystem_slug` and `title_slug` in branch names.**  
   Replace spaces with `-` and avoid underscores or uppercase letters.

10. **Do not rename or manually duplicate `conftest.py` if generated by CI setup.**  
    `conftest.py` is auto-committed by `ci_workflow.ensure()` for `src/` imports. Treat it as generated project infrastructure and preserve its expected filename.

## Class and Function Naming

1. **Name Python classes in PascalCase.**  
   Follow the TRD examples:
   - `ConsensusEngine`
   - `BuildPipeline`

2. **Name Python functions and methods in snake_case.**  
   Example:
   - `validate_write_path(...)`

3. **Name constants in UPPER_SNAKE_CASE.**  
   Follow the existing constant pattern:
   - `GENERATION_SYSTEM`
   - `SWIFT_GENERATION_SYSTEM`
   - `UI_ADDENDUM`

4. **Use noun-based names for classes that encapsulate system roles.**  
   Prefer names like `ConsensusEngine` and `BuildPipeline`, not verb phrases like `BuildingManager` or `DoConsensus`.

5. **Use verb-based names for functions that perform actions or checks.**  
   Prefer names like `validate_write_path`, `ensure_workflow`, or `build_project`.

6. **Keep naming aligned between module and primary class.**  
   If a module exists primarily to hold one main class, the module name must reflect that class’s domain:
   - `consensus.py` → `ConsensusEngine`
   - `build_director.py` → `BuildPipeline`

## Error and Exception Patterns

1. **Validate every user-supplied write path before any write operation.**  
   Before opening, creating, overwriting, moving, or generating a file from a user-controlled path, call:
   ```python
   from path_security import validate_write_path
   safe_path = validate_write_path(user_supplied_path)
   ```
   Use `safe_path` for the write operation.

2. **Never write to a raw user-supplied path.**  
   Do not pass `user_supplied_path` directly to `open()`, `Path.write_text()`, `Path.write_bytes()`, rename operations, archive extraction targets, or generator outputs.

3. **Use the validated path result even when traversal is suspected.**  
   `validate_write_path(...)` returns a safe default on traversal. Continue using the returned safe path rather than attempting to “fix” the original path manually.

4. **Centralize write-path validation through `path_security.validate_write_path`.**  
   Do not implement ad hoc traversal checks with string matching, `..` filtering, or custom normalization in feature code.

5. **Apply path validation to generated files as well as direct user uploads.**  
   If a path is derived from user input indirectly, validate it before writing.

## Import and Module Organisation

1. **Import `validate_write_path` directly from `path_security` in any module that performs writes from external input.**  
   Use:
   ```python
   from path_security import validate_write_path
   ```

2. **Keep import paths compatible with `src/`-based project layout.**  
   Organize imports so modules under `src/` are imported consistently through the configured test/runtime environment supported by generated `conftest.py`.

3. **Do not create alternate CI workflow filenames for the same platform role.**  
   If workflow logic is updated, modify `crafted-ci.yml` or `crafted-ci-macos.yml` instead of introducing parallel variants like `ci.yml`, `macos.yml`, or `crafted-ci-swift.yml`.

4. **Keep backend responsibilities separated by module domain.**  
   Place consensus-related logic in `src/consensus.py` and build-pipeline orchestration logic in `src/build_director.py` unless a new responsibility clearly warrants a new module.

## Comment and Documentation Rules

1. **Classify documentation-related PRs using documentation keywords only when matching the documented keyword list behavior.**  
   For logic that detects docs PRs, use title/metadata matching consistent with the documented pattern:
   - include keywords such as `"naming convention"`, `"glossary"`, `"changelog"`
   - evaluate against lowercased PR title text

2. **Do not reintroduce removed keyword behavior without explicit need.**  
   The TRD notes the v5.0 keyword list was removed in v6.0. If touching docs-PR classification code, preserve current behavior unless the change explicitly restores and documents that logic.

3. **Use comments to record enforced identifier formats exactly where they are applied.**  
   For accessibility identifier generation, include the format near the implementation:
   ```swift
   // Convention: {module}-{component}-{role}-{context?}
   ```

4. **Document generated or infrastructure-managed files inline when relevant.**  
   When referencing `conftest.py` behavior in tooling or contributor docs, state that it is auto-committed by `ci_workflow.ensure()` for `src/` import support.

## CraftedApp-Specific Patterns

1. **Assign an accessibility identifier to every interactive SwiftUI element.**  
   Apply `.accessibilityIdentifier(...)` to all buttons, fields, rows, and other interactive controls.

2. **Use the CraftedApp accessibility identifier format exactly.**  
   All interactive-element identifiers must follow:
   `{module}-{component}-{role}-{context?}`

3. **Use lowercase kebab-case segments in accessibility identifiers.**  
   Each non-dynamic segment must be lowercase and hyphen-separated, for example:
   - `auth-touchid-button`
   - `settings-anthropic-key-field`

4. **Append dynamic context values at the end of the accessibility identifier.**  
   For item-specific controls, place the dynamic identifier in the final segment position, for example:
   - `navigator-project-row-{projectId}`
   - `stream-gate-card-{gateId}`
   - `stream-gate-yes-button-{gateId}`

5. **Keep module and component names stable across related controls.**  
   Controls in the same feature area must share the same leading module/component segments. Example:
   - `settings-anthropic-key-field`
   - `settings-anthropic-key-test-button`
   - `settings-anthropic-key-reveal-button`

6. **Use role terms that describe the interactive element type.**  
   Valid role examples from the TRD include:
   - `button`
   - `field`
   - `row`
   - `card`

7. **Do not omit the dynamic context suffix when multiple rendered controls are data-bound.**  
   For lists, streams, navigator rows, and gate controls, include the item identifier to keep selectors unique.

8. **Preserve exact established accessibility identifier prefixes when extending a feature.**  
   Extend existing patterns rather than inventing alternatives. For example, continue using:
   - `auth-*`
   - `settings-anthropic-key-*`
   - `navigator-project-*`
   - `stream-gate-*`

9. **When adding new stream gate actions, keep action naming in the component segment and `button` as the role segment.**  
   Follow the existing pattern:
   - `stream-gate-yes-button-{gateId}`
   - `stream-gate-skip-button-{gateId}`
   - `stream-gate-stop-button-{gateId}`

10. **When adding new Python generation or build orchestration logic, place system prompt constants alongside the owning engine module.**  
    Follow the established `src/consensus.py` pattern for constants such as:
    - `GENERATION_SYSTEM`
    - `SWIFT_GENERATION_SYSTEM`
    - `UI_ADDENDUM`

11. **When adding build pipeline orchestration logic, prefer extending `BuildPipeline` in `src/build_director.py` before creating parallel orchestrator types.**  
    New pipeline stages or coordination behavior should be added to the existing orchestration domain unless a separate responsibility is clearly required.
