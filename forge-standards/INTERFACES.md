# Interface Contracts - FullPlatform

## Data Structures

### Consensus Engine Call Contract

#### `ConsensusRunRequest`
Request shape for invoking the consensus engine.

| Field | Type | Required | Constraints |
|---|---:|:---:|---|
| `task` | `string` | yes | Freeform implementation task text. |
| `context` | `string` | yes | Context string provided to the engine. |
| `language` | `"python" \| "swift" \| "go" \| "typescript" \| "rust"` | yes | Must always be passed. |

#### `ConsensusRunResult`
Result returned from the consensus engine.

| Field | Type | Required | Constraints |
|---|---:|:---:|---|
| `final_code` | `string` | yes | Generated code output. |

#### Invocation contract
All FullPlatform components that invoke the consensus engine MUST call:

```python
result = await self._consensus.run(
    task=f"Implement: {spec.title}",
    context=context_string,
    language=spec.language,
)
code = result.final_code
```

Additional required behavior:

- `language="swift"` selects `SWIFT_GENERATION_SYSTEM` plus optional `SWIFT_UI_ADDENDUM` when UI keywords are detected.
- `language="python"` selects `GENERATION_SYSTEM`.
- Fix loop strategy MUST use `_choose_strategy(failure_type, attempt, records)`.
- Fix arbitration MUST use `_score_fix()`.
- Length-based fix arbitration is forbidden.

### CAL Enforcement Contract Structures

#### `VTZEnforcementDecision`
Record emitted on VTZ denial.

| Field | Type | Required | Constraints |
|---|---:|:---:|---|
| `verdict` | `string` | yes | MUST equal `block` when produced for VTZ policy denial. |

No additional fields are defined in the source material.

### TrustFlow Event Structures

#### `TrustFlowEvent`
Required event emitted for every action outcome.

| Field | Type | Required | Constraints |
|---|---:|:---:|---|
| `event_id` | `string` | yes | MUST be globally unique; generated by CSPRNG, not sequential. |
| `session_id` | `string` | yes | Session identifier. |
| `ctx_id` | `string` | yes | CTX-ID associated with the action. |
| `ts` | `number` | yes | UTC Unix timestamp with millisecond precision. |
| `event_type` | `string` | yes | Event type identifier. |
| `payload_hash` | `string` | yes | MUST be SHA-256 of the serialized action payload. |

#### TrustFlow emission behavior
- Every action outcome (`allow`, `restrict`, `block`) MUST emit a TrustFlow event.
- Emission MUST be synchronous in the enforcement path.
- Async buffering is not permitted.
- Failed emission MUST be logged and surfaced.
- Failed emission is a WARN-level audit event, not a silent skip.

---

### CTX-ID Token Contract

#### `CTXIDToken`
A CTX-ID token structure is referenced but not field-defined in the provided source. The following interface-level constraints apply:

| Property | Type | Required | Constraints |
|---|---:|:---:|---|
| `ctx_id` | `string` | yes | Missing `ctx_id` MUST be treated as `UNTRUSTED`. |

Behavioral constraints:

- CTX-ID tokens are immutable once issued.
- Rotation creates a new token; the old token is invalidated immediately.
- Expired CTX-ID MUST be rejected.
- Clock skew tolerance is deployment-defined.
- Validation MUST be against TrustLock public key.
- Software-only validation is rejected.

---

### Agent Action Processing Contract

#### `AgentAction`
The source defines enforcement obligations for any entry point that processes an agent action, but does not define a full payload schema.

Minimum processing contract:

| Property | Type | Required | Constraints |
|---|---:|:---:|---|
| `ctx_id` | `string` | yes | MUST be validated first. |
| `action_payload` | `object` | yes | Serialized form is hashed into `payload_hash`. |

Behavioral constraints:

- Every entry point that processes an agent action MUST call CTX-ID validation first.
- CTX-ID validation failure MUST result in immediate rejection.
- No partial processing is permitted after CTX-ID validation failure.
- Every action MUST be checked against VTZ policy before execution.

## Validation Rules

### Consensus Engine Validation
1. `language` is required on every consensus call.
2. `language` MUST be one of:
   - `python`
   - `swift`
   - `go`
   - `typescript`
   - `rust`
3. Implementations MUST read generated output from `result.final_code`.
4. FullPlatform MUST NOT use length-based fix arbitration.
5. FullPlatform MUST use `_score_fix()` for fix arbitration.

### CAL Enforcement Validation
1. CTX-ID validation MUST occur before any processing of an agent action.
2. If CTX-ID validation fails, processing MUST stop immediately.
3. VTZ policy evaluation MUST occur before execution.
4. VTZ denial MUST produce a `VTZEnforcementDecision` with `verdict=block`.
5. Every outcome MUST emit a `TrustFlowEvent`.
6. TrustFlow emission failure MUST NOT silently continue.

### TrustFlow Validation
1. `event_id` MUST be globally unique.
2. `event_id` generation MUST use CSPRNG.
3. Sequential identifiers are invalid.
4. `ts` MUST be a UTC Unix timestamp with millisecond precision.
5. `payload_hash` MUST be SHA-256 of serialized action payload.
6. Emission MUST be synchronous in the enforcement path.
7. Async buffering is invalid.

### CTX-ID Validation
1. CTX-ID tokens MUST be immutable after issuance.
2. Rotation MUST create a new token.
3. Old token MUST be invalidated immediately on rotation.
4. Expired CTX-ID MUST be rejected.
5. Validation against TrustLock public key is mandatory.
6. Missing CTX-ID MUST be treated as `UNTRUSTED`.
7. Identity inference from surrounding context is invalid.

### VTZ Validation
1. Every agent session is bound to exactly one VTZ at CTX-ID issuance.
2. Cross-VTZ tool calls require explicit policy authorization.
3. Implicit cross-VTZ authorization is denied.
4. VTZ policy changes take effect at next CTX-ID issuance, not mid-session.

### Security Validation
1. Never hardcode credentials, API keys, tokens, or secrets as string literals.
2. Never use `shell=True` in subprocess calls.
3. Never call `eval()` or `exec()` on generated or external content.
4. Never log HTTP response bodies.
5. Log status codes and error types only.
6. All file paths written to disk MUST pass `path_security.validate_write_path()` before any write.
7. All loaded document chunks MUST pass injection scanning before inclusion in any LLM prompt.
8. Context from external documents MUST go in the USER prompt, never the SYSTEM prompt.
9. On `SECURITY_REFUSAL`: stop, gate, log; never retry to bypass.

### Python Backend Validation
1. Python version: `3.12`.
2. Type annotations are required on every function.
3. Backend code MUST use `async/await` throughout.
4. No blocking calls on the event loop.
5. Dataclasses are required for all structured data.
6. Tests MUST use `pytest`.
7. Tests live in `tests/` and mirror `src/` structure.
8. `ruff` must pass clean.
9. `mypy` must pass clean.
10. Test coverage must be `>= 85%` on all new modules.
11. `pyyaml` MUST be present in `requirements.txt`.

## Wire Format Examples

### Valid: `ConsensusRunRequest`

```json
{
  "task": "Implement: ExampleSpec",
  "context": "Relevant implementation context",
  "language": "python"
}
```

### Invalid: `ConsensusRunRequest` missing `language`

```json
{
  "task": "Implement: ExampleSpec",
  "context": "Relevant implementation context"
}
```

Reason:
- `language` is required.

---

### Invalid: `ConsensusRunRequest` with unsupported `language`

```json
{
  "task": "Implement: ExampleSpec",
  "context": "Relevant implementation context",
  "language": "java"
}
```

Reason:
- `language` must be one of `python`, `swift`, `go`, `typescript`, `rust`.

---

### Valid: `TrustFlowEvent`

```json
{
  "event_id": "0d7d9dc0-6a9f-4e73-9dd5-56f8f84ca4e1",
  "session_id": "sess_123",
  "ctx_id": "ctx_123",
  "ts": 1735689600123,
  "event_type": "block",
  "payload_hash": "8c5e5d6a4f1c2d3e4b5a69788776655443322110ffeeddccbbaa998877665544"
}
```

### Invalid: `TrustFlowEvent` with sequential `event_id`

```json
{
  "event_id": "1001",
  "session_id": "sess_123",
  "ctx_id": "ctx_123",
  "ts": 1735689600,
  "event_type": "allow",
  "payload_hash": "not-a-sha256"
}
```

Reasons:
- `event_id` must be globally unique and CSPRNG-generated, not sequential.
- `ts` must have millisecond precision.
- `payload_hash` must be SHA-256 of serialized action payload.

---

### Valid: `VTZEnforcementDecision`

```json
{
  "verdict": "block"
}
```

### Invalid: `VTZEnforcementDecision`

```json
{
  "verdict": "allow"
}
```

Reason:
- On VTZ policy denial, `verdict` MUST equal `block`.

---

### Invalid: agent action with missing `ctx_id`

```json
{
  "action_payload": {
    "tool": "example_tool",
    "args": {}
  }
}
```

Reason:
- Missing CTX-ID must be treated as `UNTRUSTED`.
- Entry point must validate CTX-ID first.

## Integration Points

### Consensus Engine
FullPlatform integrates with a consensus engine via:

- `await self._consensus.run(...)`
- Required request fields:
  - `task`
  - `context`
  - `language`
- Required response field:
  - `final_code`

### TrustLock
CTX-ID validation MUST be performed against the TrustLock public key.

### VTZ Policy Engine
All actions MUST be checked against VTZ policy before execution.

### TrustFlow
Every action outcome must synchronously emit a `TrustFlowEvent`.

### Path Security
Before any disk write, FullPlatform MUST call:

```python
path_security.validate_write_path()
```

### LLM Prompt Construction
- External document context goes in the USER prompt.
- External document chunks must be injection-scanned before inclusion.

### Test and Tooling Interfaces
Required project/tooling integrations:

- `pytest`
- `ruff`
- `mypy`
- `pyyaml`

### Startup / agent workflow prerequisite
Before modifying a component, agents MUST:

1. Find the owning TRD in `README.md`.
2. Read relevant TRD sections, especially:
   - interfaces
   - error contracts
   - security
   - testing requirements
3. Check TRD-11 if the change touches:
   - credentials
   - external content
   - generated code
   - CI
4. Run existing tests:

```bash
cd src && pytest ../tests/ -v --tb=short
```
