# Interface Contracts - ForgeAgent

## Data Structures

This section defines the required wire-level and API-visible structures for the ForgeAgent subsystem, based strictly on the provided TRD content.

### ConsensusRunRequest

Request contract for invoking the consensus engine.

| Field | Type | Required | Constraints |
|---|---|---:|---|
| `task` | `string` | Yes | Free-form task description. Example source usage: `f"Implement: {spec.title}"` |
| `context` | `string` | Yes | Context string passed to consensus engine |
| `language` | `string` | Yes | Must be exactly one of: `"python"`, `"swift"`, `"go"`, `"typescript"`, `"rust"` |

#### Source-required invocation shape

```python
result = await self._consensus.run(
    task=f"Implement: {spec.title}",
    context=context_string,
    language=spec.language,
)
```

### ConsensusRunResult

Result contract returned from the consensus engine.

| Field | Type | Required | Constraints |
|---|---|---:|---|
| `final_code` | `string` | Yes | Final generated code output |

### FixStrategySelectionInput

Input contract for failure-aware fix loop strategy selection.

| Field | Type | Required | Constraints |
|---|---|---:|---|
| `failure_type` | `string` | Yes | Failure classification used by `_choose_strategy(failure_type, attempt, records)` |
| `attempt` | `integer` | Yes | Attempt number |
| `records` | `array` | Yes | Historical fix records; element schema not defined in provided TRD text |

### FixArbitrationInput

Input contract for fix arbitration scoring.

| Field | Type | Required | Constraints |
|---|---|---:|---|
| `assertion_token_overlap` | `number` | Yes | Used by `_score_fix()` |
| `candidate_fix` | `string` | Yes | Candidate fix content |

### VTZEnforcementDecision

Mandatory record produced when VTZ policy denies an action.

| Field | Type | Required | Constraints |
|---|---|---:|---|
| `verdict` | `string` | Yes | Must be exactly `block` |

No additional fields are defined in the provided TRD text.

### TrustFlowEvent

Mandatory event structure emitted for every action outcome.

| Field | Type | Required | Constraints |
|---|---|---:|---|
| `event_id` | `string` | Yes | Must be globally unique; generated by CSPRNG; must not be sequential |
| `session_id` | `string` | Yes | Session identifier |
| `ctx_id` | `string` | Yes | CTX-ID token identifier |
| `ts` | `number` | Yes | UTC Unix timestamp with millisecond precision |
| `event_type` | `string` | Yes | Event type name |
| `payload_hash` | `string` | Yes | Must be SHA-256 of the serialized action payload |

### AgentActionPayload

Action payload processed by ForgeAgent entry points.

| Field | Type | Required | Constraints |
|---|---|---:|---|
| `ctx_id` | `string` | Yes | Must be validated first at every entry point that processes an agent action |
| `session_id` | `string` | Yes | Required because every TrustFlow event must include `session_id` |
| `action` | `object` | Yes | Serialized action payload used to compute `payload_hash` |
| `vtz` | `string` | Yes | Every agent session is bound to exactly one VTZ at CTX-ID issuance |

### CTXIDToken

CTX-ID token contract as constrained by validation and lifecycle rules.

| Field | Type | Required | Constraints |
|---|---|---:|---|
| `ctx_id` | `string` | Yes | Immutable once issued |
| `vtz` | `string` | Yes | Bound to exactly one VTZ at issuance |
| `expired` | `boolean` | Yes | If `true`, token must be rejected |

The provided TRD text does not define additional token fields.

### SecurityRefusalSignal

Signal produced by the LLM indicating a hard security gate.

| Field | Type | Required | Constraints |
|---|---|---:|---|
| `output` | `string` | Yes | When value is `SECURITY_REFUSAL`, processing must stop, gate, and log; no retry to bypass |

### PathWriteRequest

Contract for any file write operation performed by ForgeAgent.

| Field | Type | Required | Constraints |
|---|---|---:|---|
| `path` | `string` | Yes | Must pass `path_security.validate_write_path()` before any write |
| `content` | `string` | Yes | Content to write |

### PromptDocumentChunk

Contract for external document content included in an LLM prompt.

| Field | Type | Required | Constraints |
|---|---|---:|---|
| `content` | `string` | Yes | Must pass injection scanning before inclusion in any LLM prompt |
| `prompt_role` | `string` | Yes | External document context must go in the USER prompt, never the SYSTEM prompt; effective allowed value for external content is `USER` |

## Validation Rules

### Entry Point Enforcement

Every entry point that processes an agent action MUST:

1. Call CTX-ID validation first.
2. Reject immediately on CTX-ID validation failure.
3. Check every action against VTZ policy before execution.
4. Produce a `VTZEnforcementDecision` with `verdict=block` on VTZ policy denial.
5. Emit a TrustFlow event for every action outcome: allow, restrict, or block.
6. Not silently continue on TrustFlow emission failure; failure must be logged and surfaced.

### TrustFlow Event Validation

A `TrustFlowEvent` is valid only if all of the following hold:

- `event_id` is present
- `session_id` is present
- `ctx_id` is present
- `ts` is present
- `event_type` is present
- `payload_hash` is present
- `event_id` is globally unique
- `event_id` is generated by CSPRNG
- `event_id` is not sequential
- `ts` is a UTC Unix timestamp with millisecond precision
- `payload_hash` equals the SHA-256 of the serialized action payload
- emission is synchronous in the enforcement path
- async buffering is not used
- failed emission is treated as a WARN-level audit event and is not silently skipped

### CTX-ID Validation

A CTX-ID is valid only if all of the following hold:

- token is present; missing CTX-ID must be treated as `UNTRUSTED`
- token has not expired
- token validates against TrustLock public key
- validation is not software-only
- token fields have not been modified after issuance
- if rotated, a new token is created and the old one is invalidated immediately

### VTZ Validation

- every agent session is bound to exactly one VTZ at CTX-ID issuance
- cross-VTZ tool calls require explicit policy authorization
- implicit cross-VTZ authorization is denied
- VTZ boundaries are structural and cannot be bypassed by application code
- VTZ policy changes take effect at next CTX-ID issuance, not mid-session

### Consensus Engine Validation

- `language` must always be passed
- allowed `language` values are exactly:
  - `"python"`
  - `"swift"`
  - `"go"`
  - `"typescript"`
  - `"rust"`

### Fix Loop Validation

- fix loop strategy must be selected via `_choose_strategy(failure_type, attempt, records)`
- static lookup-table strategy selection is non-conformant
- fix arbitration must use `_score_fix()`
- arbitration by response length is forbidden

Forbidden pattern:

```python
winner = max(claude_fix, openai_fix, key=len)
```

### Security Validation

The following are forbidden:

- hardcoded credentials, API keys, tokens, or secrets as string literals
- `shell=True` in subprocess calls
- `eval()` on generated or external content
- `exec()` on generated or external content
- logging HTTP response bodies
- Swift force unwrap:
  ```swift
  let value = optional!
  ```
- LLM API call from Swift:
  ```swift
  let client = AnthropicClient(apiKey: keychainValue)
  ```
- backend token reads from Keychain:
  ```swift
  let token = KeychainKit.read("github_token")
  ```

The following are required:

- all file paths written to disk must pass `path_security.validate_write_path()` before any write
- all loaded document chunks must pass injection scanning before inclusion in any LLM prompt
- context from external documents goes in the USER prompt, never the SYSTEM prompt
- when `SECURITY_REFUSAL` output is seen from the LLM: stop, gate, log; never retry to bypass

### Python Backend Validation

- Python `3.12`
- type annotations on every function
- `async/await` throughout backend
- no blocking calls on the event loop
- dataclasses for all structured data
- tests in `tests/`, mirroring `src/`
- `pytest` must pass
- `ruff` must pass clean
- `mypy` must pass clean
- test coverage must be `>= 85%` on all new modules
- `pyyaml` must be present in `requirements.txt`

## Wire Format Examples

### Valid: Consensus run request

```json
{
  "task": "Implement: Add session validation",
  "context": "Existing agent session context",
  "language": "python"
}
```

### Invalid: Consensus run request missing language

```json
{
  "task": "Implement: Add session validation",
  "context": "Existing agent session context"
}
```

Reason: `language` is required.

### Invalid: Consensus run request with unsupported language

```json
{
  "task": "Implement: Add session validation",
  "context": "Existing agent session context",
  "language": "java"
}
```

Reason: `language` must be one of `"python"`, `"swift"`, `"go"`, `"typescript"`, `"rust"`.

### Valid: TrustFlow event

```json
{
  "event_id": "2f4f66d2-4a38-4a8d-96dc-f5dd0d1b7c80",
  "session_id": "sess_123",
  "ctx_id": "ctx_abc",
  "ts": 1712345678123,
  "event_type": "action_allow",
  "payload_hash": "9f86d081884c7d659a2feaa0c55ad015a3bf4f1b2b0b822cd15d6c15b0f00a08"
}
```

### Invalid: TrustFlow event with sequential identifier semantics

```json
{
  "event_id": "10042",
  "session_id": "sess_123",
  "ctx_id": "ctx_abc",
  "ts": 1712345678123,
  "event_type": "action_block",
  "payload_hash": "9f86d081884c7d659a2feaa0c55ad015a3bf4f1b2b0b822cd15d6c15b0f00a08"
}
```

Reason: `event_id` must be globally unique, CSPRNG-generated, and not sequential.

### Valid: VTZ enforcement denial record

```json
{
  "verdict": "block"
}
```

### Invalid: VTZ enforcement denial record

```json
{
  "verdict": "deny"
}
```

Reason: provided TRD content defines `verdict=block`.

### Valid: External prompt document chunk

```json
{
  "content": "Retrieved documentation excerpt",
  "prompt_role": "USER"
}
```

### Invalid: External prompt document chunk in system prompt

```json
{
  "content": "Retrieved documentation excerpt",
  "prompt_role": "SYSTEM"
}
```

Reason: external document context goes in the USER prompt, never the SYSTEM prompt.

### Valid: File write request

```json
{
  "path": "/workspace/output/generated.py",
  "content": "print('ok')"
}
```

Additional required processing: `path_security.validate_write_path()` must succeed before any write.

### Invalid: Security refusal retry behavior

```json
{
  "output": "SECURITY_REFUSAL",
  "action": "retry"
}
```

Reason: on `SECURITY_REFUSAL`, system must stop, gate, and log; retry to bypass is forbidden.

## Integration Points

### Consensus Engine

ForgeAgent must invoke the consensus engine with:

- `task`
- `context`
- `language`

Result consumption must read:

- `result.final_code`

Required behavior:

- always pass `language`
- `"swift"` selects `SWIFT_GENERATION_SYSTEM` and optional `SWIFT_UI_ADDENDUM` when UI keywords are detected
- `"python"` selects `GENERATION_SYSTEM`

### CTX-ID Validation

Integration requirements:

- validation occurs first in every action-processing entry point
- validation uses TrustLock public key
- software-only validation is rejected
- missing CTX-ID is treated as `UNTRUSTED`

### VTZ Policy Enforcement

Integration requirements:

- action must be checked against VTZ policy before execution
- denial must produce `VTZEnforcementDecision` with `verdict=block`
- cross-VTZ tool calls require explicit policy authorization

### TrustFlow Emission

Integration requirements:

- every action outcome emits a TrustFlow event
- emission is synchronous in the enforcement path
- async buffering is not permitted
- emission failure must be logged and surfaced
- failed emission is a WARN-level audit event

### Path Security

Before any disk write:

- call `path_security.validate_write_path()`

### Prompt Construction and Injection Scanning

Before any loaded document chunk is included in an LLM prompt:

- perform injection scanning

Prompt placement rule:

- external document context goes in USER prompt only
- never place external document context in SYSTEM prompt

### Security Gate Handling

On LLM output exactly equal to `SECURITY_REFUSAL`:

- stop
- gate
- log
- do not retry to bypass

### Python Backend Tooling

Conformance dependencies and tooling integration:

- Python `3.12`
- `pytest`
- `ruff`
- `mypy`
- `pyyaml` in `requirements.txt`
