# Interface Contracts - ForgeAgent

## Data Structures

This document defines the wire-format and API contract requirements for the ForgeAgent subsystem based on the provided TRD excerpts and Forge component standards.

### `ConsensusRunRequest`

Request passed to the consensus engine.

| Field | Type | Required | Constraints |
|---|---|---:|---|
| `task` | `string` | Yes | Human-readable implementation task. Example source usage: `f"Implement: {spec.title}"` |
| `context` | `string` | Yes | Context string supplied to consensus engine |
| `language` | `"python" \| "swift" \| "go" \| "typescript" \| "rust"` | Yes | Must always be passed; omission is non-conformant |

#### Behavioral contract
- Must be invoked as:
  - `result = await self._consensus.run(task=..., context=..., language=...)`
- `language="swift"` selects `SWIFT_GENERATION_SYSTEM` plus optional `SWIFT_UI_ADDENDUM` when UI keywords are detected.
- `language="python"` selects `GENERATION_SYSTEM`.

---

### `ConsensusRunResult`

Result returned by the consensus engine.

| Field | Type | Required | Constraints |
|---|---|---:|---|
| `final_code` | `string` | Yes | Final code output selected by the consensus process |

---

### `VTZEnforcementDecision`

Record produced when VTZ policy denies an action.

| Field | Type | Required | Constraints |
|---|---|---:|---|
| `verdict` | `string` | Yes | Must be exactly `block` when produced for VTZ policy denial |

#### Behavioral contract
- VTZ policy denial MUST produce a `VTZEnforcementDecision` record with `verdict=block`.

---

### `TrustFlowEvent`

Audit/enforcement event emitted for every action outcome.

| Field | Type | Required | Constraints |
|---|---|---:|---|
| `event_id` | `string` | Yes | Must be globally unique; generated by CSPRNG; must not be sequential |
| `session_id` | `string` | Yes | Session identifier |
| `ctx_id` | `string` | Yes | CTX-ID associated with the action |
| `ts` | `number` | Yes | UTC Unix timestamp with millisecond precision |
| `event_type` | `string` | Yes | Event type identifier |
| `payload_hash` | `string` | Yes | Must be SHA-256 of the serialized action payload |

#### Behavioral contract
- Every action outcome (`allow`, `restrict`, `block`) MUST emit a TrustFlow event.
- Emission MUST be synchronous in the enforcement path.
- Async buffering is not permitted.
- Emission failure MUST NOT silently continue.
- Failed emission is a WARN-level audit event.

---

### `CTXIDToken`

Immutable token used for agent session trust and isolation.

| Field | Type | Required | Constraints |
|---|---|---:|---|
| `ctx_id` | `string` | Yes | Token identifier; immutable once issued |

#### Behavioral contract
- CTX-ID tokens are immutable once issued.
- Rotation creates a new token.
- Old token is invalidated immediately upon rotation.
- Expired CTX-ID must be rejected.
- Missing CTX-ID must be treated as `UNTRUSTED`.
- CTX-ID must be validated against TrustLock public key.
- Software-only validation is rejected.

> Note: Additional CTX-ID token fields are not defined in the provided source excerpt and therefore are not specified here.

---

### `ForgeAgentActionPayload`

Serialized action payload used in enforcement and TrustFlow hashing.

| Field | Type | Required | Constraints |
|---|---|---:|---|
| `payload` | `object` | Yes | Must be serialized deterministically before SHA-256 hashing for `payload_hash` generation |

#### Behavioral contract
- Every entry point that processes an agent action MUST validate CTX-ID first.
- Every action MUST be checked against VTZ policy before execution.

---

### `SecurityRefusalSignal`

Special model output indicating security refusal.

| Field | Type | Required | Constraints |
|---|---|---:|---|
| `output` | `string` | Yes | When equal to `SECURITY_REFUSAL`, processing must stop |

#### Behavioral contract
- On `SECURITY_REFUSAL`: stop, gate, log.
- Never retry to bypass.

## Validation Rules

## 1. Entry-point enforcement

Every ForgeAgent entry point that processes an agent action MUST perform, in this order:

1. CTX-ID validation first
2. Immediate rejection on CTX-ID validation failure
3. VTZ policy check before execution
4. Produce `VTZEnforcementDecision` with `verdict=block` on VTZ denial
5. Emit `TrustFlowEvent` for every outcome

### Rejection requirements
- No partial processing is permitted after CTX-ID validation failure.
- Missing CTX-ID is treated as `UNTRUSTED`.
- Expired CTX-ID must be rejected.

---

## 2. TrustFlow validation

A `TrustFlowEvent` is valid only if all of the following are true:

- `event_id` is present
- `session_id` is present
- `ctx_id` is present
- `ts` is present
- `event_type` is present
- `payload_hash` is present
- `event_id` is globally unique
- `event_id` is generated using CSPRNG
- `event_id` is not sequential
- `ts` is UTC Unix timestamp with millisecond precision
- `payload_hash` is SHA-256 of the serialized action payload

### Emission rules
- Emission must be synchronous in the enforcement path.
- Async buffering is invalid.
- Emission failure must be logged and surfaced.
- Emission failure must not be silently skipped.

---

## 3. CTX-ID validation

A CTX-ID is valid only if all of the following are true:

- Token is present, unless the caller is intentionally handled as `UNTRUSTED`
- Token is not expired
- Token has not been invalidated by rotation
- Token is validated against TrustLock public key

### Invalid CTX-ID conditions
- Missing token
- Expired token
- Rotated-out token
- Software-only validation path

### Rotation rules
- Rotation creates a new token.
- Old token becomes invalid immediately.
- VTZ policy changes take effect at next CTX-ID issuance, not mid-session.

---

## 4. VTZ enforcement

- Every agent session is bound to exactly one VTZ at CTX-ID issuance.
- Cross-VTZ tool calls require explicit policy authorization.
- Implicit authorization is denied.
- VTZ boundaries are structural and cannot be bypassed by application code.

---

## 5. Consensus engine invocation

Consensus engine requests are valid only if:

- `language` is always passed
- `language` is one of:
  - `python`
  - `swift`
  - `go`
  - `typescript`
  - `rust`

### Selection behavior
- `swift` -> `SWIFT_GENERATION_SYSTEM` + optional `SWIFT_UI_ADDENDUM`
- `python` -> `GENERATION_SYSTEM`

### Result usage
- Code must be read from `result.final_code`

---

## 6. Fix loop arbitration

### Required
- Strategy selection must use `_choose_strategy(failure_type, attempt, records)`
- Fix arbitration must use `_score_fix()`
- `_score_fix()` is based on assertion token overlap

### Forbidden
- Static lookup table for failure strategy
- Length-based arbitration of candidate fixes

Invalid example:

```python
winner = max(claude_fix, openai_fix, key=len)
```

---

## 7. Security rules

### Credentials and secrets
- Never hardcode credentials, API keys, tokens, or secrets as string literals.

### Subprocess
- Never use `shell=True` in subprocess calls.

### Dynamic execution
- Never call `eval()` or `exec()` on generated or external content.

### Logging
- Never log HTTP response bodies.
- Log status codes and error types only.

### File writes
- All file paths written to disk must pass `path_security.validate_write_path()` before any write.

### External document prompt inclusion
- All loaded document chunks must pass injection scanning before inclusion in any LLM prompt.
- Context from external documents goes in the USER prompt, never the SYSTEM prompt.

### Security refusal handling
- When output is `SECURITY_REFUSAL`: stop, gate, log.
- Never retry to bypass.

## Integration Points

### Invocation contract

```python
result = await self._consensus.run(
    task=f"Implement: {spec.title}",
    context=context_string,
    language=spec.language,   # "python" | "swift" | "go" | "typescript" | "rust"
```

- `language` must always be provided.

## CAL enforcement path

ForgeAgent must integrate the following in every action-processing entry point:

1. CTX-ID validation
2. VTZ policy evaluation
3. Action execution only if permitted
4. `VTZEnforcementDecision` creation on denial
5. `TrustFlowEvent` emission for all outcomes

## Non-conformance summary

A ForgeAgent implementation is non-conformant if it does any of the following:

- Processes agent actions before CTX-ID validation
- Continues partially after CTX-ID validation failure
- Executes actions before VTZ policy check
- Fails to emit TrustFlow events for `allow`, `restrict`, or `block`
- Silently skips failed TrustFlow emission
- Omits `language` in consensus requests
- Uses length-based fix arbitration
- Uses software-only CTX-ID validation
- Permits implicit cross-VTZ access
- Retries after `SECURITY_REFUSAL`
- Writes files without `path_security.validate_write_path()`
- Logs HTTP response bodies
- Uses `shell=True`
- Uses `eval()` or `exec()` on generated or external content
