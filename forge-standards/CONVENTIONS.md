# Code Conventions

## File and Directory Naming

1. **Place all Python implementation files for ForgeAgent under `src/` using lowercase snake_case filenames.**
   - Valid examples:
     - `src/consensus.py`
     - `src/build_director.py`
   - Do not use CamelCase, hyphens, or mixed-case filenames for Python modules.

2. **Use the established subsystem directory names exactly as defined by the architecture standards.**
   - Allowed canonical directories:
     - `src/cal/`
     - `src/dtl/`
     - `src/trustflow/`
     - `src/vtz/`
     - `src/trustlock/`
     - `src/mcp/`
     - `src/rewind/`
     - `sdk/connector/`
   - Do not invent alternate abbreviations or rename these directories.

3. **Mirror `src/` structure exactly under `tests/`.**
   - If implementation code is in `src/mcp/policy_engine.py`, tests must live under `tests/mcp/`.
   - Keep subsystem alignment exact; do not place tests in unrelated top-level folders.

4. **Keep required repository-root files at the repo root with no directory prefix.**
   - These filenames must be committed at root:
     - `README.md`
     - `CODEOWNERS`
     - `LICENSE`
     - `.gitignore`
     - `.editorconfig`
     - `VERSION`
     - `Makefile`
     - `pyproject.toml`
   - Do not move these into `docs/`, `.github/`, or any other subdirectory.

5. **Use the exact CI workflow filenames when adding or updating CI workflows.**
   - Ubuntu workflow: `crafted-ci.yml`
   - macOS Swift workflow: `crafted-ci-macos.yml`
   - Do not create renamed variants for the same purpose.

6. **Do not manually relocate or rename `conftest.py` created by `ci_workflow.ensure()`.**
   - Treat it as required support for `src/` imports in CI.
   - If CI setup requires `conftest.py`, keep it committed in the location generated by the workflow utility.

7. **Use descriptive subsystem filenames that map directly to the primary class or responsibility.**
   - Follow the architecture examples:
     - `src/consensus.py` for `ConsensusEngine`
     - `src/build_director.py` for `BuildPipeline`
   - Avoid vague names like `helpers.py`, `utils.py`, `misc.py`, or `common.py`.

## Class and Function Naming

1. **Name Python classes in PascalCase.**
   - Valid examples:
     - `ConsensusEngine`
     - `BuildPipeline`

2. **Name Python functions and methods in snake_case.**
   - Valid examples:
     - `validate_write_path`
     - `ensure_ci_workflow`
     - `build_release_artifacts`

3. **Name constants in UPPER_SNAKE_CASE.**
   - Valid examples:
     - `GENERATION_SYSTEM`
     - `SWIFT_GENERATION_SYSTEM`
     - `UI_ADDENDUM`

4. **Use names that reflect a single subsystem responsibility.**
   - A file named `consensus.py` should expose consensus-related types and logic.
   - A file named `build_director.py` should expose build pipeline orchestration logic.
   - Do not place unrelated classes into a module just because they are small.

5. **Name accessibility identifiers using the exact pattern `{module}-{component}-{role}-{context?}`.**
   - Apply `.accessibilityIdentifier()` to **all interactive elements**.
   - Valid examples:
     - `auth-touchid-button`
     - `auth-passcode-button`
     - `settings-anthropic-key-field`
     - `settings-anthropic-key-test-button`
     - `settings-anthropic-key-reveal-button`
     - `navigator-project-row-{projectId}`
     - `stream-gate-card-{gateId}`
     - `stream-gate-yes-button-{gateId}`
     - `stream-gate-skip-button-{gateId}`
     - `stream-gate-stop-button-{gateId}`

6. **When an accessibility identifier targets a specific entity, append the entity identifier as the final segment.**
   - Use forms like:
     - `navigator-project-row-{projectId}`
     - `stream-gate-yes-button-{gateId}`
   - Do not insert IDs in the middle of the identifier.

7. **Use the mandatory branch naming format exactly as specified.**
   - Format:
     - `forge-agent/build/{engineer_id}/{subsystem_slug}/pr-{N:03d}-{title_slug}`
   - Example:
     - `forge-agent/build/e123/mcp/pr-042-policy-cache`
   - Do not substitute another prefix for `forge-agent`.

## Error and Exception Patterns

1. **Validate paths before any write operation that uses a user-supplied path.**
   - Required pattern:
     ```python
     from path_security import validate_write_path

     safe_path = validate_write_path(user_supplied_path)
     ```
   - Apply this before creating, overwriting, appending, moving, or exporting files.

2. **Use the validated path returned by `validate_write_path(...)` for the actual write.**
   - Do not validate a path and then write using the original untrusted variable.
   - Correct pattern:
     ```python
     safe_path = validate_write_path(user_supplied_path)
     with open(safe_path, "w", encoding="utf-8") as f:
         f.write(content)
     ```

3. **Treat every externally influenced filesystem destination as untrusted until validated.**
   - This includes paths from:
     - user input
     - CLI arguments
     - API payloads
     - environment variables
     - config files
     - branch-derived or title-derived filenames

4. **Rely on `validate_write_path()` to normalize traversal attempts rather than implementing ad hoc traversal filtering.**
   - Do not replace it with custom string checks like:
     - rejecting only `".."` substrings
     - manually stripping slashes
     - regex-only sanitizers

5. **Apply path validation before any code path that may indirectly trigger a write.**
   - If a helper function, exporter, or artifact generator writes to disk, validate before invoking the write path.
   - Do not defer validation until after temporary path construction if the destination is user-controlled.

## Import and Module Organisation

1. **Import `validate_write_path` from `path_security` wherever write-path validation is required.**
   - Required import:
     ```python
     from path_security import validate_write_path
     ```

2. **Keep modules aligned to the subsystem directories defined by the architecture.**
   - Examples:
     - Conversation abstraction code goes in `src/cal/`
     - Data trust label code goes in `src/dtl/`
     - TrustFlow audit logic goes in `src/trustflow/`
     - VTZ enforcement goes in `src/vtz/`
     - MCP policy logic goes in `src/mcp/`

3. **Keep SDK connector code under `sdk/connector/` only.**
   - Do not place connector SDK implementation in `src/` unless it is application runtime code rather than SDK surface.

4. **Do not use test-only directory layouts that diverge from production module layout.**
   - `tests/<subsystem>/` must correspond to `src/<subsystem>/`.
   - Preserve module naming symmetry to keep imports and ownership obvious.

5. **Keep CI-related support files consistent with the expected workflow structure.**
   - If code depends on CI bootstrap imports, preserve the `conftest.py` arrangement produced by `ci_workflow.ensure()`.

## Comment and Documentation Rules

1. **Document naming-sensitive patterns using the exact canonical wording from the TRDs when applicable.**
   - For accessibility identifiers, reference the pattern exactly:
     - `{module}-{component}-{role}-{context?}`
   - Do not redefine it with alternate segment names.

2. **When documenting branch creation, use the exact mandatory branch format.**
   - Always show:
     - `forge-agent/build/{engineer_id}/{subsystem_slug}/pr-{N:03d}-{title_slug}`

3. **When documenting path-write safety, include the required validation pattern before any sample write.**
   - Every write example that accepts an external path must show:
     ```python
     from path_security import validate_write_path
     safe_path = validate_write_path(user_supplied_path)
     ```

4. **Do not classify pull requests as documentation PRs using the removed v5.0 keyword list.**
   - Do not recreate or reference `_docs_keywords`.
   - Do not implement logic based on title keywords such as `"naming convention"`, `"glossary"`, or `"changelog"` from that removed list.

5. **Keep comments concrete and tied to subsystem behavior, naming, or safety constraints.**
   - Explain why a path is validated, why an identifier includes an entity ID, or why a file must live at repo root.
   - Do not add generic comments that restate the code without a project-specific rule.

## ForgeAgent-Specific Patterns

1. **For any interactive macOS Swift UI element, always set `.accessibilityIdentifier()` explicitly.**
   - This is mandatory for buttons, fields, rows, cards, and other interactive controls.
   - Do not rely on generated or implicit identifiers.

2. **Build accessibility identifiers from ordered segments: module, component, role, optional context.**
   - Examples:
     - `settings-anthropic-key-field`
     - `settings-anthropic-key-test-button`
   - Keep segment order stable across the subsystem.

3. **Use entity-scoped accessibility identifiers for repeated UI elements in lists, streams, or cards.**
   - Examples:
     - `navigator-project-row-{projectId}`
     - `stream-gate-card-{gateId}`
     - `stream-gate-stop-button-{gateId}`

4. **When implementing ForgeAgent build and orchestration code, preserve the established module-to-type mapping.**
   - `src/consensus.py` owns `ConsensusEngine` and related generation-system constants.
   - `src/build_director.py` owns `BuildPipeline`.
   - Extend these files only with directly related logic.

5. **Use subsystem slugs in branch names that match the architecture directory names whenever applicable.**
   - Preferred examples:
     - `cal`
     - `dtl`
     - `trustflow`
     - `vtz`
     - `trustlock`
     - `mcp`
     - `rewind`
   - Avoid branch slugs that obscure subsystem ownership.

6. **When generating files from ForgeAgent workflows, validate any destination path before writing generated artifacts.**
   - This applies to:
     - code generation
     - export commands
     - CI-generated support files
     - replay or audit outputs

7. **Keep root-level governance and build files as basename-stable integration points.**
   - If ForgeAgent creates or updates `README.md`, `CODEOWNERS`, `LICENSE`, `.gitignore`, `.editorconfig`, `VERSION`, `Makefile`, or `pyproject.toml`, write them at repo root and preserve those exact basenames.
