# Code Conventions

## File and Directory Naming

1. **Place ForgeAgent backend Python modules under `src/`.**
   - Use flat module files in `src/` for core orchestration components.
   - Required existing naming patterns:
     - `src/consensus.py`
     - `src/build_director.py`
     - `src/github_tools.py`
     - `src/build_ledger.py`
     - `src/document_store.py`
     - `src/ci_workflow.py`

2. **Use the prescribed subsystem directories exactly as defined by Forge standards.**
   - New code for these areas must be placed only in:
     - `src/cal/`
     - `src/dtl/`
     - `src/trustflow/`
     - `src/vtz/`
     - `src/trustlock/`
     - `src/mcp/`
     - `src/rewind/`
     - `sdk/connector/`

3. **Mirror `src/` structure exactly under `tests/`.**
   - If implementation file is `src/build_director.py`, test file must be in `tests/build_director/` or equivalent mirrored placement for that module.
   - If implementation file is `src/mcp/policy_engine.py`, tests must live under `tests/mcp/`.

4. **Name Python files in `snake_case.py`.**
   - Match the examples from the architecture documents.
   - Do not use hyphens, spaces, or CamelCase in Python filenames.

5. **Keep CI workflow filenames fixed.**
   - Ubuntu workflow file must be named `crafted-ci.yml`.
   - macOS Swift workflow file must be named `crafted-ci-macos.yml`.

6. **Do not rename or duplicate `conftest.py` generated by CI setup.**
   - `conftest.py` is auto-committed by `ci_workflow.ensure()` to support `src/` imports.
   - If code depends on test import bootstrapping, use the generated `conftest.py` instead of creating alternate import shims.

7. **Use the mandatory branch naming format exactly.**
   - Branch names must be:
     - `forge-agent/build/{engineer_id}/{subsystem_slug}/pr-{N:03d}-{title_slug}`
   - Keep the literal prefix `forge-agent` unchanged for compatibility.
   - Example:
     - `forge-agent/build/alex/document-store/pr-014-vector-refresh`

8. **Use lowercase kebab-case for branch path segments other than `engineer_id`.**
   - `subsystem_slug` and `title_slug` must use lowercase letters, digits, and hyphens only.
   - `pr-{N:03d}` must always be zero-padded to 3 digits.

## Class and Function Naming

1. **Name Python classes in `PascalCase`.**
   - Follow the established class names from the TRDs:
     - `ConsensusEngine`
     - `BuildPipeline`
     - `GitHubTool`
     - `WebhookReceiver`
     - `BuildLedger`
     - `DocumentStore`

2. **Name Python functions and methods in `snake_case`.**
   - Follow existing examples:
     - `chunk()`
     - `embed()`
     - `retrieve()`
     - `ensure()`
     - `validate_write_path()`

3. **Name module-level constants in `UPPER_SNAKE_CASE`.**
   - Follow established constant names:
     - `GENERATION_SYSTEM`
     - `SWIFT_GENERATION_SYSTEM`
     - `UI_ADDENDUM`

4. **Name accessibility identifiers using this exact pattern:**
   - `{module}-{component}-{role}-{context?}`
   - Apply this to all interactive macOS Swift UI elements via `.accessibilityIdentifier()`.

5. **Keep all accessibility identifier segments lowercase and hyphen-separated.**
   - Valid examples:
     - `auth-touchid-button`
     - `auth-passcode-button`
     - `settings-anthropic-key-field`
     - `settings-anthropic-key-test-button`
     - `settings-anthropic-key-reveal-button`

6. **When an interactive UI element is bound to a model/entity, append the entity identifier as the final segment.**
   - Examples:
     - `navigator-project-row-{projectId}`
     - `stream-gate-card-{gateId}`
     - `stream-gate-yes-button-{gateId}`
     - `stream-gate-skip-button-{gateId}`
     - `stream-gate-stop-button-{gateId}`

7. **Use semantic role terms in accessibility identifiers.**
   - Preferred role segment values are concrete UI roles such as:
     - `button`
     - `field`
     - `row`
     - `card`
   - Do not use generic endings like `item`, `thing`, or `control` when a concrete role exists.

## Error and Exception Patterns

1. **Validate every user-supplied write path before any filesystem write.**
   - This is mandatory for all file creation, overwrite, append, move, extract, or generated artifact writes.

2. **Use `validate_write_path()` directly before the write operation.**
   - Required pattern:
   ```python
   from path_security import validate_write_path

   safe_path = validate_write_path(user_supplied_path)
   ```
   - Write only to `safe_path`.

3. **Do not write to the original user-supplied path after validation.**
   - Once validation is performed, discard the untrusted path variable for the write call.

4. **Treat path traversal defense as required even when the path appears internal.**
   - If any path value originated from request data, webhook payloads, PR metadata, CLI args, environment configuration, or generated user-controlled content, validate it.

5. **Rely on `validate_write_path()` returning a safe default on traversal attempts.**
   - Do not reintroduce fallback logic that bypasses the validated result.

6. **When wrapping lower-level write failures, preserve the validated path in logs or exception context, not the original untrusted path.**
   - This prevents error handling from accidentally reusing unsafe inputs.

## Import and Module Organisation

1. **Import path validation from the canonical module only.**
   - Use:
   ```python
   from path_security import validate_write_path
   ```
   - Do not create local aliases or duplicate helper implementations for write-path validation.

2. **Keep implementation modules under `src/` importable by the generated test configuration.**
   - Assume `conftest.py` created by `ci_workflow.ensure()` is the supported mechanism for `src/` imports during testing.

3. **Add CI workflow management logic to `src/ci_workflow.py`.**
   - Code that creates, updates, or ensures `crafted-ci.yml`, `crafted-ci-macos.yml`, or test import bootstrap files belongs in this module.

4. **Keep GitHub integration code in `src/github_tools.py`.**
   - Tooling for GitHub operations or webhook handling must be implemented in that module or its mirrored subsystem location.

5. **Keep build orchestration logic in `src/build_director.py`.**
   - PR type routing, confidence gates, and pipeline orchestration belong there.

6. **Keep ledger state and claim/release/heartbeat logic in `src/build_ledger.py`.**
   - Do not spread build ownership state transitions across unrelated modules.

7. **Keep document retrieval/chunking/embedding code in `src/document_store.py`.**
   - Retrieval pipeline logic must stay co-located with `DocumentStore`.

8. **Keep consensus prompt/system definitions in `src/consensus.py`.**
   - `GENERATION_SYSTEM`, `SWIFT_GENERATION_SYSTEM`, and `UI_ADDENDUM` must remain module-level definitions in that area.

## Comment and Documentation Rules

1. **Document naming rules using the phrase “naming convention” when adding rule-oriented docs or comments.**
   - This matches the wording used by pipeline keyword classification.

2. **Use the exact phrase “glossary” for glossary documents and “changelog” for change history documents.**
   - Do not invent alternate labels if the content is meant to be detected or classified by title/metadata rules.

3. **When writing PR titles or documentation-only labels that should be recognized as docs-related, prefer the known keyword set from the pipeline logic.**
   - Include terms such as:
     - `naming convention`
     - `glossary`
     - `changelog`

4. **For accessibility identifiers in Swift UI code, place the identifier inline on the interactive element with `.accessibilityIdentifier()`.**
   - Do not document the identifier only in comments while omitting the actual modifier.

5. **Use comments to record why a path is safe only when the write path was derived from untrusted input.**
   - The comment must reference the actual validation call site, not a generic statement like “safe path”.

6. **Do not use comments as a substitute for structural compliance.**
   - Examples:
     - Do not comment that a path is validated if `validate_write_path()` is not called.
     - Do not comment an expected accessibility identifier if `.accessibilityIdentifier()` is absent.

## ForgeAgent-Specific Patterns

1. **Before generating or updating CI files, route the operation through `ci_workflow.ensure()`.**
   - This includes creation or maintenance of:
     - `crafted-ci.yml`
     - `crafted-ci-macos.yml`
     - generated `conftest.py`

2. **Keep ForgeAgent branch creation compatible with the fixed `forge-agent` prefix.**
   - Even if subsystem naming evolves, do not alter the branch root segment.

3. **Use subsystem slugs in branch names that match the implementation area.**
   - Examples:
     - `build-director`
     - `document-store`
     - `github-tools`
     - `ci-workflow`
     - `mcp`
     - `trustflow`

4. **When adding macOS Swift interactive UI, every interactive control must have an accessibility identifier.**
   - Apply `.accessibilityIdentifier()` to buttons, fields, rows, cards, and equivalent tappable/focusable controls.

5. **For repeated dynamic UI collections, include the backing entity identifier in the accessibility identifier.**
   - This is required for rows, cards, and action buttons tied to project or gate records.

6. **When implementing write operations for build artifacts, generated files, workflow files, or replay outputs, validate the destination path in the same function that performs the write.**
   - Do not rely on callers to pre-validate.

7. **Place replay engine code under `src/rewind/`.**
   - Any Forge Rewind functionality must use that directory.

8. **Place policy engine code under `src/mcp/`.**
   - MCP Policy Engine functionality must use that directory.

9. **Place conversation abstraction code under `src/cal/`.**
   - CAL functionality must not be mixed into unrelated top-level modules.

10. **Place trust and enforcement components in their assigned directories only.**
    - `src/dtl/` for Data Trust Label components
    - `src/trustflow/` for TrustFlow audit stream components
    - `src/vtz/` for Virtual Trust Zone enforcement
    - `src/trustlock/` for cryptographic machine identity components

11. **When adding tests for a subsystem directory, mirror that subsystem directory name exactly.**
    - Examples:
      - `src/rewind/...` → `tests/rewind/...`
      - `src/trustflow/...` → `tests/trustflow/...`
      - `src/cal/...` → `tests/cal/...`

12. **Use existing canonical class names when extending established modules.**
    - Extend `BuildLedger` in `src/build_ledger.py`, not a parallel class with overlapping responsibility.
    - Extend `DocumentStore` in `src/document_store.py`, not a duplicate retrieval/store abstraction.
    - Extend `GitHubTool` or `WebhookReceiver` in `src/github_tools.py` for GitHub-facing behavior.
