# DECISIONS.md

## [Place ConsensusDevAgent implementation in `src/` and keep design artifacts in `forge-docs/`]
**Status:** Accepted  
**Context:** The repository structure explicitly separates source artifacts from implementation. `forge-docs/` is the source of truth for TRDs and PRDs, while backend runtime code lives in `src/`. The subsystem must fit the established repository contract so that planning, implementation, review, and audit tooling can locate artifacts deterministically.  
**Decision:** Implement the ConsensusDevAgent subsystem under `src/`. Store no runtime implementation code in `forge-docs/`. Treat `forge-docs/` as read-before-build design input only, and keep any subsystem-specific architecture constraints mirrored in `forge-standards/` when they become repository-wide conventions.  
**Consequences:** Implementation, tests, and imports must assume `src/` as the Python backend root. Design-time documents remain separate from executable code, which preserves the documentation-to-implementation boundary and simplifies build pipeline discovery. Any code generator, reviewer, or file writer in this subsystem must not emit runtime files into `forge-docs/`.  
**Rejected alternatives:** Keeping subsystem code beside TRDs in `forge-docs/` was rejected because it mixes source-of-truth documents with executable artifacts and breaks repository layout expectations. Creating a new top-level directory for ConsensusDevAgent was rejected because the repository already defines `src/` as the Python backend implementation location.

## [Treat build pipeline stage transitions as irreversible gates]
**Status:** Accepted  
**Context:** The build pipeline defines ordered stages from scope through merge, and the requirements explicitly state there is no undo on gate decisions. This must be documented as an implementation constraint, not left as an operator convention, because ConsensusDevAgent participates in generation, review, and gating behavior.  
**Decision:** Enforce all ConsensusDevAgent gate decisions as append-only and irreversible. Do not implement any undo, rollback, reopen, or silent bypass mechanism for a completed gate decision. If a prior gate outcome must be superseded, create a new explicit decision event or restart from the appropriate earlier workflow entrypoint rather than mutating history.  
**Consequences:** Audit logs, state machines, and operator UX must model gate outcomes as immutable historical facts. The subsystem must preserve a monotonic decision trail, which supports compliance and post hoc review. Recovery flows must be expressed as new actions, not edits to prior gate records.  
**Rejected alternatives:** Allowing operators to undo a gate decision was rejected because it would weaken auditability and contradict the stated pipeline rule. Soft-undo via hidden state mutation was rejected because it preserves the appearance of compliance while corrupting the decision history. Deleting prior gate records was rejected because it destroys the audit trail.

## [Integrate ConsensusDevAgent as a stage-aware participant in the existing build pipeline]
**Status:** Accepted  
**Context:** The repository README defines a fixed pipeline: Scope, PRD Plan, PRD Gen, PR Plan, Code Gen, 3-Pass Review, and merge completion flow. ConsensusDevAgent must operate within this pipeline rather than creating an alternate orchestration model, because stage contracts and escalation paths already exist in TRD-3.  
**Decision:** Implement ConsensusDevAgent as a subsystem that consumes and emits stage-bounded artifacts compatible with the existing build pipeline. Do not introduce a parallel lifecycle outside the repository’s defined stages. Respect stage ordering, gate semantics, and escalation points defined by the pipeline contracts.  
**Consequences:** The subsystem must expose clear handoffs for planning, generation, arbitration, review, and gating within the established sequence. State transitions must be attributable to a pipeline stage. Any consensus logic must be stage-scoped and must not skip mandatory review or gate checkpoints.  
**Rejected alternatives:** Defining a standalone ConsensusDevAgent workflow independent of the build pipeline was rejected because it would fragment orchestration and bypass existing controls. Allowing dynamic stage reordering by the subsystem was rejected because it conflicts with documented stage contracts and weakens predictability.

## [Use consensus and arbitration during generation, but keep final progression gated by the pipeline]
**Status:** Accepted  
**Context:** The build pipeline specifies parallel generation with arbitration and model comparison in generation phases. ConsensusDevAgent exists to coordinate multi-agent output, but coordination does not replace the gate structure or review obligations.  
**Decision:** Use consensus, comparison, and arbitration to produce candidate outputs during planning and code generation stages, but require that final progression to the next stage occur only through the defined pipeline gate. Treat consensus as an input to decisions, not as an implicit approval to proceed.  
**Consequences:** The subsystem may aggregate outputs from multiple agents or models, but it must still emit a single stage artifact and await the next formal gate. Consensus records should be auditable and separable from approval records. This prevents generation-time agreement from bypassing review, security, or operator controls.  
**Rejected alternatives:** Treating majority agreement among agents as automatic stage approval was rejected because it bypasses gate semantics. Letting arbitration output directly merge or advance work without review was rejected because the pipeline requires explicit downstream checks, especially the 3-pass review.

## [Escalate on `SECURITY_REFUSAL` and never auto-bypass]
**Status:** Accepted  
**Context:** Mandatory security controls require `SECURITY_REFUSAL` behavior in generation prompts and outputs. The requirement is explicit: if `SECURITY_REFUSAL` appears in output, stop the PR, gate, log, and never auto-bypass. ConsensusDevAgent must honor this even when multiple agents disagree or one output appears otherwise usable.  
**Decision:** When any participating generation or review output produces `SECURITY_REFUSAL`, immediately halt progression for the affected work item, create a gate event for operator review, and record the incident in the audit trail. Do not auto-retry around the refusal, suppress it, merge alternative outputs silently, or continue execution as though consensus can override the refusal.  
**Consequences:** Security refusal becomes a hard stop condition in subsystem orchestration. Arbitration logic must treat refusal as dominant over ordinary candidate selection. Operators may review and initiate subsequent actions, but the subsystem itself must not bypass or downgrade the refusal.  
**Rejected alternatives:** Ignoring a refusal if another model produced acceptable output was rejected because mandatory controls state that refusal must stop the PR. Automatic retry with modified prompts to evade the refusal was rejected because it constitutes an auto-bypass. Downgrading refusal to a warning was rejected because it weakens the required security posture.

## [Validate every write path before file output]
**Status:** Accepted  
**Context:** Mandatory security controls require `path_security.validate_write_path()` on every file path before write. ConsensusDevAgent will generate, reconcile, and materialize artifacts, so it must enforce path validation uniformly across all write operations.  
**Decision:** Invoke `path_security.validate_write_path()` for every filesystem write performed by ConsensusDevAgent, including generated code, temporary artifacts promoted to durable outputs, logs written to repository paths, and any consensus-produced patch materialization. Do not permit direct writes that bypass this validation layer.  
**Consequences:** All file-emitting components must route through a shared write abstraction or otherwise prove validation occurs first. This constrains implementation toward centralized file I/O policy enforcement and reduces path traversal or repository boundary violations. Review and test coverage must include validation-path enforcement.  
**Rejected alternatives:** Validating only user-supplied paths was rejected because generated and computed paths can also be unsafe. Performing ad hoc path checks per caller was rejected because it is error-prone and hard to audit. Trusting internal components without validation was rejected because mandatory controls require validation on every write.

## [Run generated outputs through mandatory 3-pass review, including security]
**Status:** Accepted  
**Context:** The pipeline mandates a 3-pass review sequence: correctness, performance, and security. Security controls further require that all generated code pass security review including bandit. ConsensusDevAgent may improve candidate quality, but it does not eliminate the need for mandatory review passes.  
**Decision:** Require all ConsensusDevAgent-generated code and code-affecting artifacts to pass the full 3-pass review pipeline, with security review treated as mandatory and non-skippable. Do not mark consensus-selected output as production-ready prior to completion of these review passes.  
**Consequences:** The subsystem must produce outputs in a form review tooling can consume and must preserve sufficient provenance for reviewers to assess generated changes. Performance and security concerns discovered after consensus remain blocking issues. Consensus selection is therefore necessary but not sufficient for acceptance.  
**Rejected alternatives:** Skipping review for high-confidence consensus outputs was rejected because it violates the pipeline. Running only security review and omitting correctness or performance was rejected because all three passes are required. Treating bandit as optional was rejected because the security controls explicitly require it.

## [Preserve explicit audit trail records for consensus, gating, and escalation events]
**Status:** Accepted  
**Context:** The build pipeline references audit trail requirements and error escalation schema. The explicit no-undo rule makes historical traceability especially important. Because ConsensusDevAgent introduces multi-agent coordination and arbitration, its intermediate and final decisions must be reviewable after the fact.  
**Decision:** Record consensus inputs, arbitration outcomes, gate decisions, security escalations, and operator-required interruptions as explicit audit trail events. Store them as append-only records tied to the relevant pipeline stage and work item. Do not collapse intermediate decision points into only the final artifact.  
**Consequences:** The subsystem must carry event metadata sufficient to explain why a candidate was selected, blocked, or escalated. This increases implementation discipline around provenance and observability. Incident investigation, compliance review, and debugging become possible without reconstructing hidden in-memory state.  
**Rejected alternatives:** Logging only final outputs was rejected because it obscures the rationale behind consensus and gate outcomes. Keeping audit data only in transient process memory was rejected because it cannot support later review. Overwriting prior event records with “latest state” snapshots was rejected because it violates append-only auditability.

## [Treat untrusted review content as labeled input, not trusted instruction]
**Status:** Accepted  
**Context:** Mandatory security controls state that PR review comments must be labeled `[UNTRUSTED REVIEWER COMMENT]` in prompts, and flagged chunks used in generation require gating to the operator before proceeding. ConsensusDevAgent will likely ingest external review material during coordination and revision.  
**Decision:** Represent reviewer-supplied or otherwise untrusted external content as explicitly labeled untrusted input in any prompt or decision context used by ConsensusDevAgent. Do not treat such content as authoritative system instruction. If flagged untrusted content is used in generation, require a gate card to the operator before proceeding.  
**Consequences:** Prompt assembly, consensus scoring, and revision workflows must preserve trust boundaries. The subsystem must differentiate repository truth sources from external commentary and enforce operator review when flagged material influences generation. This reduces prompt injection risk and prevents reviewer text from silently steering implementation.  
**Rejected alternatives:** Treating PR comments as trusted implementation instructions was rejected because the security controls explicitly classify them as untrusted. Stripping labels before prompt construction was rejected because it removes a critical trust boundary signal. Allowing flagged chunks to influence generation without operator gating was rejected because it violates the mandatory control.
